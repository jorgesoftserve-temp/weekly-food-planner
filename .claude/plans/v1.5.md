# v1.5 — Post-MVP Enhancements & Agentic Tooling (shipped / in-flight tail)

> **Status: shipped, with a small in-flight tail (menu MCP server wiring).** Retrospective
> plan-of-record for the enhancement wave between the v1 MVP demo (`0b24aab`) and the start of the
> **[v2](./v2.md)** epic. These are refinements *within* the v1 lifecycle (still ending at accept) —
> they hardened the menu/grocery flow, added flexibility, and stood up the agentic/MCP developer
> tooling that v2 now builds on. Source: commits `b1b33e1` → `b6f8186` (+ untracked `apps/menu-mcp-server/`).

## Context

The MVP proved the deterministic engine and the core lifecycle. v1.5 closed the gaps that surfaced
in real use: menus needed to be more than fixed 7-day shared weeks, grocery needed to reflect who's
actually eating and how much, and the team needed repeatable agentic scaffolding (sub-agents, skills,
MCP servers) to build v2 safely against the determinism contract. No engine invariant changed —
every addition routes through the existing `inputs_hash` / `accepted_seed` boundary.

## Shipped — grouped by theme

### Menu flexibility
- **Variable duration + custom mode + clone-from-history** (`f90112b`) — menus are no longer fixed-length; a menu can be cloned from a prior one via a picker.
- **Ongoing-week filter + better error reporting** (`0633034`) — generation respects the current week's progress; clearer failure surfaces.
- **Add-slot on a draft** (`f9a3860`) — append a slot to a draft before accepting.

### Participants & per-menu overrides
- **Per-menu participants + meal-frequency overrides** (`2eda10c`) — choose who participates in a given menu and override their meal frequency for that menu (overlay → `inputs_hash`).
- **Default `meal_frequency` per `age_category` on member insert** (`be2749b`) — sensible defaults so new members are immediately plannable.

### Grocery correctness
- **Unified draft + accept grocery path** through `recomputeGroceryListsForMenu` (`5203942`) — one recompute function for both states (the boundary v2 must keep inventory-agnostic).
- **Per-member buckets + freshness-aware purchase days** (`df9c1a2`).
- **Servings-aware grocery scaling** (`b778cee`).
- **Shop-for-subset filter** + **filtered markdown/CSV exports honoring `?shop_for=`** (`f9a3860`, `c274492`) — the presentation-only `applyShopForFilter` that v2's all-members view + menu member-switcher are modeled on.

## Invariants preserved

Nothing in v1.5 touched the engine's determinism contract: per-menu participant/frequency overrides
and the shop-for subset are **overlay / presentation-only**; the unified grocery recompute is the
single boundary v2 keeps inventory- and engine-agnostic. This is the baseline the **[v2](./v2.md)**
epic extends past accept.
