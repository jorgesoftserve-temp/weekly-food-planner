## Part A — Cook mode (the flagship v1.9 feature)

**Today:** `menu_slots` is `(id, menu_id, day_of_week, meal_key, meal_type, recipe_id, target_member_id)`
with `UNIQUE NULLS NOT DISTINCT` — **no completion state**. The `recipe-cook-mock` is fully interactive
(checkable ingredients + numbered steps, "Mark as cooked" enabled once every step is checked) but is
flagged *"Proposed product addition — not yet in any PRD."* v1.9 makes it real.

### A1. Schema (→ `supabase-migration-author`)
- Migration `…_tbl_alter_menu_slots_add_cooked_at.sql`: `ALTER TABLE public.menu_slots ADD COLUMN
  cooked_at TIMESTAMPTZ NULL;` + `COMMENT`. NULL = not cooked; timestamp = when marked cooked.
- **No new RLS** — `menu_slots` is already gated through its parent `menus` workspace policies; confirm
  the existing UPDATE policy covers a member toggling `cooked_at` (it should; verify via `supabase-local`).
- `pnpm run gen:types`; extend `MenuSlotRecord` (it flows through `menu_slots` selects in the menu module).
- **Determinism guard:** `cooked_at` is *runtime* state, never an engine input. It must NOT enter the
  constraint-engine inputs hash or any generation path (engine purity rule #9). Document this in the
  migration comment and the engine CLAUDE if needed.

### A2. Data layer + route (→ `supabase-module-author` then `route-handler-engineer`)
- A `markSlotCooked` / `unmarkSlotCooked` mutation (sets/clears `cooked_at = now()` server-side — the
  client never sends the timestamp). Prefer a route handler / server action so the clock lives on the
  server (NOT in the engine).
- React hook `useMarkSlotCooked` invalidating the active-menu query.

### A3. UI — promote `recipe-cook-mock` (→ `ui-component-builder`)
- Decide the surface: a focused **Cook view** reached from a menu SlotCard ("Cook" action) — either a
  route `/menu/cook/[slotId]` or a full-screen Sheet. Reuse `@/lib/recipe-icon`, the recipe's real
  ingredients + instructions (from the recipe module), and the success-tint completion treatment.
- "Mark as cooked" calls `useMarkSlotCooked`; cooked slots render a success check in `MenuView`.
- **Cooked-state affordance in `MenuView`:** a cooked slot gets a subtle success ring/badge (token-only).

### A4. Dashboard stat (→ `ui-component-builder`)
- Add a **"Cooked this week"** `StatCard` (e.g. `N of M meals cooked`) computed from `cooked_at` over
  the active menu's slots. Mirrors the existing day/grocery stat cards.

## Part B — Global search (cross-module)

**Today:** there is **no `/search` route**. The `search-mock` is an Airbnb-style segmented query builder
(module picker + per-module filters + keyword), flagged *"New functionality; mock only."*

- New route `apps/web/app/(app)/search/page.tsx` + `_components/` (segmented bar, results list). Add the
  nav entry + the topbar search affordance (`design-lab/_components/topbar-search.tsx` is the lab analog).
- **Scope decision (confirm with user before building):** start with **Recipes search** (name/cuisine/tag/
  meal filters over the existing recipes query) as the first module, with the segmented bar designed to
  extend to menu/grocery/members later — rather than building all four modules at once. Avoids a heavy
  cross-entity query layer in a "mini" release.
- Reuse `recipe-card` for results. No schema change for the recipes module path.
