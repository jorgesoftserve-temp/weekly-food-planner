# v1 — MVP: Deterministic Weekly Menu Planner (shipped)

> **Status: shipped.** This is a retrospective plan-of-record for the v1 MVP — the release the
> 26-Jun-2026 MVP boundary refers to. It captures what was built and the architectural decisions
> that still constrain every later release. Source of truth: [`docs/PRD/OVERVIEW_PRD.md`](../../docs/PRD/OVERVIEW_PRD.md) §5–§8,
> the constraint engine, and commits `0caa04e` (initial) → `0b24aab` (MVP 1 demo).

## Context

The product's core innovation is a **deterministic, constraint-based weekly menu generator** that
respects per-member dietary restrictions and age groups, and produces reproducible output. v1
delivered the full lifecycle from auth to grocery export, ending at **accept** (post-accept
execution/pantry is v2). Everything later is an extension of the invariants set here.

## Scope delivered (OVERVIEW §6 "Included in MVP")

- **Authentication** — registration, email verification, login/logout, password reset, group ownership (Supabase Auth).
- **Group / household management** — workspaces with member roles (`creator`, `admin`, `member`), dietary profiles, calorie goals, meal-frequency routines.
- **Recipe management** — recipe CRUD with images, ingredient tracking, dietary tagging, shared vs member-specific meals; in-app catalog ingredient additions.
- **Menu generation** — 7-day planning, hard-constraint validation, **deterministic** generation, shared + member-specific slots.
- **Menu & grocery viewing** — in-app week/day menu view, in-app grocery view (shared + per-member), PDF-ready layout.
- **Grocery list generation** — aggregated ingredients, freshness-aware grouping, shared + member-specific items.
- **Draft / accept** — generate → review draft → accept; grocery list recomputed from the accepted menu.
- **Export** — markdown + CSV export of menu/grocery.
- **Quality gates** — Vitest unit + integration, CI workflow, Dockerized local Supabase stack.
- **Settings** — account (display name, password), dietary-preference + meal-schedule editors.

## Architecture invariants established (still binding in v1.5 / v2 / v3)

- **Determinism contract** — *same inputs + same seed = same output*. `accepted_seed = SHA-256(inputs_hash + sorted slot recipe-tuples {day_of_week, meal_key, target_member_id, recipe_id})`; `inputs_hash = SHA-256(canonical JSON of the full engine input)`. ([`apps/web/lib/api/menu-accept.ts`](../../apps/web/lib/api/menu-accept.ts))
- **Engine purity** — [`packages/constraint-engine`](../../packages/constraint-engine/) is pure TS: no `Date.now()`, `new Date()`, `Math.random()`, `crypto.randomUUID()`; seeded RNG only; inputs/outputs JSON-round-trip. Slot enumeration → hard filter (`filter.ts`) → greedy assignment (`assign.ts`).
- **Constraint safety** — hard dietary restrictions / allergies / excluded ingredients are *never* violated; meal-type match is a hard filter; calorie balancing is the only soft consideration.
- **Three Supabase clients** — browser / server / admin; never `@supabase/auth-helpers-nextjs`.
- **RLS-first + soft delete** — `workspaces`, `workspace_members`, `recipes`, `menus` carry `is_deleted`; read policies filter it; unique constraints are partial. Role checks via `fn_user_workspace_role`.
- **Per-menu constraint overlay** — generation-time overlay over member profiles with **silent additive dedup** (agent-log 07–08), feeding `inputs_hash`.
- **RO-RO + conventions** — Receive-Object/Return-Object, fat-arrow, one export per file, kebab-case files, Tailwind `flex`+`gap`, shadcn via CLI.

## Key surfaces (where it lives)

- Engine: [`packages/constraint-engine/src/`](../../packages/constraint-engine/src/) — `types.ts`, `slots.ts`, `filter.ts`, `assign.ts`, `index.ts`.
- Data layer: [`packages/supabase/`](../../packages/supabase/) — migrations (`fn_`→`enum_`→`tbl_`→`rls_`), generated + hand-written types, per-table modules + `.react.ts` hooks.
- API: [`apps/web/app/api/`](../../apps/web/app/api/) — workspaces, members, recipes, menus (generate/accept), grocery, export.
- UI: [`apps/web/app/(app)/`](../../apps/web/app/(app)/) — `dashboard`, `members`, `recipes`, `menu`, `grocery`, `settings`.

## Verification (as shipped)

## What v1 deliberately excluded (carried forward)

PDF export, AI recipe generation, nutrition APIs, real-time collaboration, budget optimization,
shopping integrations, **inventory tracking**, calendar sync. Inventory/leftovers were later promoted
into **[v2](./v2.md)** (with hosting as **[v2.3](./v2.3.md)**); community became **[v4](./v4.md)**; the AI
import + i18n became **[v3](./v3.md)**.
