> determinism rule**. This file is the buildable plan for everything that ships in v2.0.

## What v2.0 is

The coherent post-accept story — **shop → cook → leftovers → pantry-aware next week** — built entirely
from **post-accept mutable state and pure read-side views**. **Nothing here touches the constraint
engine.** That is the defining property of this release: its determinism gate collapses to "engine
inputs unchanged + grocery recompute changes *only* for a Phase-6 ingredient override." **No
golden-snapshot regen.**

Items in scope: **0** (food groups), **1** (inventory), **2** (shopping confirmation), **3** (incomplete
alerts), **4** (cook-status), **5 + 7** (leftovers + per-leftover spoilage), **9** (menu-level ingredient
substitution), **8 + 10** (all-members grocery view + per-member menu switcher).

Deferred to **[v2.1](./v2.1.md)**: items 11 + 12 (the two engine items) and item 6 + the seed-pack
manifest (community growth track).

## Part 2 — Implementation (build sequence)

Two parallel tracks inside v2.0: the **execution spine** (Phases 0→5, each depends on the prior) and the
**Track-B read/menu work** (Phases 6, 7 — depend only on accepted menus, no schema for 7). Fan these out.

### Shared data-layer ritual (every phase)
New table → `cd packages/supabase && npx supabase migration new …` (deps order: `fn_`→`enum_`→`tbl_`+idx+trg→`rls_` enable→`rls_` policies). New column → use the **`supabase-add-column`** skill. After each migration: regen `database.types.ts`, update hand-written types in `packages/supabase/src/types/db.ts`, add `module/<table>.ts` + `<table>.react.ts`, export via barrel. Route handlers call CRUD fns directly. Delegate schema to **supabase-migration-author**, route handlers to **route-handler-engineer**, UI to **ui-component-builder**, tests to **vitest-integration-author**.

### Phase 0 — Food groups (prereq for Phase 2)
- `ingredients` + `food_group text NULL`, `food_group_source` enum `{seed,ai,unset}`.
- Seed `food_group` for catalog ingredients (extend `apps/web/app/api/admin/seed-ingredients/route.ts` + `seed.sql`), `source='seed'`. **v2.0 keeps the existing inline array** — the versioned manifest is v2.1.
- `enum_metadata` label set `food_group` (via `sys_save_label`).
- `apps/web/lib/api/food-group-classify.ts` — Claude-API classify for user-created ingredients lacking a group, cache on row via `supabaseAdminClient`. Use the **claude-api** skill (prompt caching). Never in the engine.

### Phase 1 — Inventory (feature 1)
- `enum inventory_source {manual,purchase,leftover}`.
- `inventory_items` (`id, workspace_id, ingredient_id, source, quantity numeric CHECK>=0, unit, expiration_date date NULL, source_menu_id NULL, source_slot_id NULL, label text NULL, is_consumed bool, ts`). Indexes: `(workspace_id, ingredient_id) WHERE NOT is_consumed`, `(workspace_id, expiration_date) WHERE NOT is_consumed`. RLS read = active member, write = creator/admin (`fn_user_workspace_role`).
- `workspaces.leftover_max_days int NOT NULL DEFAULT 3` (`supabase-add-column`).
- Module `inventory-items.ts`/`.react.ts`. Partial-spoilage = plain `quantity` decrement (no event log in v2).
- Endpoints: `GET/POST /api/workspaces/[id]/inventory`, `PATCH/DELETE /api/workspaces/[id]/inventory/[itemId]`.
- UI feature folder `apps/web/app/(app)/inventory/` (page + `_components/`).
- **Source display tags + Menu→Pantry lifecycle.** Display labels are derived; the internal `source` enum is unchanged: `manual`→**Pantry**, `purchase`→**Menu**, `leftover`→**Leftover**. A **Menu** item (`source='purchase'` with `source_menu_id` set) reads as **Menu** only while its linked menu's week is current; once that week has ended it reads as **Pantry** (now general stock) — a **read-side / lazy derivation** (no cron; mirrors `expireLeftovers`), so **no schema change** beyond the existing `source_menu_id` + the menu's week dates. (Optional `released_to_pantry_at` only if we later want it persisted.)

### Phase 2 — Shopping confirmation + food-groups (feature 2) — depends on 0,1
- `enum shopping_status {in_progress,complete,incomplete}`, `acquired_status {pending,acquired,partial,skipped}`.
- `shopping_sessions` (`id, menu_id, workspace_id, status, completeness numeric NULL, created_by, ts`; partial unique `(menu_id) WHERE status='in_progress'`).
- `shopping_item_status` (`id, session_id, grocery_item_id, acquired_quantity, status, UNIQUE(session_id,grocery_item_id)`).
- `apps/web/lib/api/shopping-finalize.ts` — quantity-weighted `computeShoppingCompleteness`; finalize spills purchased-but-unused into `inventory_items(source='purchase')`.
- Grouping = read-side `GROUP BY ingredients.food_group` (Phase 0); **does not** touch `recomputeGroceryListsForMenu`.
- Endpoints: `POST .../menus/[menuId]/shopping-sessions`, `GET .../shopping-sessions/active?group=food_group`, `PATCH .../shopping-sessions/[sid]/items/[groceryItemId]`, `POST .../shopping-sessions/[sid]/finalize`.

### Phase 3 — Incomplete alerts (feature 3) — DERIVED, no table — depends on 2 (reads 4 if present)
- `apps/web/lib/api/menu-alerts.ts` `deriveShoppingAlertsForMenu({admin,menuId})`: missing items (from `shopping_item_status`) → reverse `ingredient→menu_slot` map (via `recipe_ingredients`, **after applying ingredient overrides from Phase 6**) → alert per not-yet-cooked slot whose shortfall isn't covered by inventory. Degrades to "all slots planned" when cook-status absent.
- Endpoint `GET /api/workspaces/[id]/menus/[menuId]/alerts`. UI = badges on the menu/slot view (`apps/web/app/(app)/menu/_components/menu-view.tsx`). Rationale for no table: state is fully derivable and high write-amplification otherwise.

### Phase 4 — Cook-status (feature 4) — depends on nothing (sequenced after 3)
- `enum slot_cook_status {planned,cooked,skipped}`.
- **`slot_completions`** (`id, menu_slot_id, workspace_id, status, cooked_at NULL, notes NULL, UNIQUE(menu_slot_id)`) — a **separate table, not a column on `menu_slots`**, so cook-status is structurally invisible to `accepted_seed` and to slot-replace paths. Absent row = `planned`. RLS via `menu_slots→menus.workspace_id` EXISTS.
- Endpoint `PATCH .../menus/[menuId]/slots/[slotId]/completion {status,notes?}`.
- **Note**: distinct from v1.9 `menu_slots.cooked_at`/`cooked_by` (cook *mode*, accepted-menu cooking toggle). `slot_completions` is the richer planned/cooked/skipped state machine that drives leftovers + alerts. Reconcile the two in the PRD (cook-mode toggle = quick mark; `slot_completions` = the execution record).
- **Cook-time ingredient reconciliation (drives the raw-ingredient leftovers in Phase 5).** Marking a slot `cooked` optionally opens a reconciliation over the slot recipe's `recipe_ingredients` (planned qty as the baseline); the user adjusts *actually used* per ingredient. Any shortfall (`planned − used`) is offered as a raw-ingredient **Pantry** leftover (Phase 5). Default = used == planned (nothing left). Purely post-accept: never touches `accepted_seed`, the engine, or `recomputeGroceryListsForMenu` — it only writes `inventory_items` rows.

### Phase 5 — Leftovers (feature 5, incl. per-leftover spoilage / item 7 + raw-ingredient leftovers) — depends on 1,4
- Two leftover kinds, both → `inventory_items` with cook provenance (`source_slot_id`, `source_menu_id`) and per-row expiry:
  - **Cooked-food surplus** — marking a slot `cooked` may emit `inventory_items(source='leftover', label, expiration_date = …)` (the prepared dish you didn't eat).
  - **Raw-ingredient leftovers (from the Phase 4 cook reconciliation)** — when `used < planned` for a recipe ingredient, emit a remainder row as **Pantry** raw stock (the ingredient you still have, e.g. "1 tomato" after using 4 of 5). Distinguish from cooked-food leftovers by provenance: add a new `inventory_source` value (e.g. `cook_remainder`, shown under the **Pantry** tag) **or** reuse `source='manual'` with `source_slot_id` set — decide at build; lean toward a dedicated `cook_remainder` value so the origin is queryable. Engine-free.
- **Per-leftover expiry (item 7)** — `expiration_date` computed **per leftover row**:
  default = `cooked_at::date + COALESCE(ingredients.max_storage_days, workspaces.leftover_max_days)`
  (reuse the existing `ingredients.max_storage_days`), then **independently editable** via `PATCH /inventory/[itemId]`. No new schema beyond the per-row `expiration_date` from Phase 1.
- Consumption: manual `is_consumed=true` (or partial decrement) **or** lazy `expireLeftovers` on inventory read (no cron in v2); expiry evaluated per row against its own `expiration_date`.

### Phase 6 — Menu-level ingredient substitution (item 9) — Track B, depends on accepted menus only
- **`menu_slot_ingredient_overrides`** (`id, menu_slot_id, workspace_id, original_ingredient_id, substitute_ingredient_id, quantity numeric NULL, unit NULL, note text NULL, created_by, ts`; `UNIQUE(menu_slot_id, original_ingredient_id)`). Keyed by `menu_slot_id` so it is **structurally unreachable from `accepted_seed`** and the engine. RLS via `menu_slots→menus.workspace_id` EXISTS, write = creator/admin.
- `recomputeGroceryListsForMenu` (`apps/web/lib/api/menu-grocery.ts`) gains an override-application step: after loading `recipe_ingredients` for the menu's slots, left-join the override map and replace `(ingredient_id, quantity?, unit?)` per slot before aggregation. **Stays inventory-agnostic and engine-agnostic** — overrides are menu state. This is the one change to recompute's inputs; cover it heavily in the recompute regression (seed unchanged, grocery changed).
- Validation: a substitute is rejected (422) if it introduces an allergen or violates an exclusive restriction for that slot's eater(s) — reuse the engine's allergen/exclusion logic (`packages/constraint-engine/src/filter.ts` `recipeViolatesAllergies` / `recipeHasExcludedIngredient`) **at the route layer** (mirrors the existing slot-replace validation in `…/slots/[slotId]/route.ts`). **This is route-layer reuse, not an engine edit — v2.0 stays engine-free.**
- Suggested substitutes come from the recipe's existing `recipe_ingredients.substitutions` JSONB catalog (currently unused) plus free choice from the ingredient catalog.
- Module `menu-slot-ingredient-overrides.ts`/`.react.ts`. Endpoint: `PUT|DELETE .../menus/[menuId]/slots/[slotId]/ingredient-overrides`. After a successful override write on an *accepted* menu, re-run `recomputeGroceryListsForMenu`.
- UI: substitution control on the slot detail / menu view (`apps/web/app/(app)/menu/_components/`), reusing the ingredient picker.

### Phase 7 — Menu/grocery view parity (items 8 + 10) — Track B, pure read-side, no schema
- **All-members aggregated grocery view (item 8)** — add a pure transform in `apps/web/lib/grocery-filter.ts`, e.g. `aggregateHouseholdGrocery({ lists })`, that unions the shared list + every per-member list into one consolidated list keyed by `(ingredient_id, unit)`, summing quantities (folding `scheduled_purchase_day` to the earliest). Composes cleanly with `applyShopForFilter` and `annotateWithInventory`. Surface as a view mode in `apps/web/app/(app)/grocery/_components/shop-for-picker.tsx` ("Everyone" / "By member" / subset). No recompute or schema change.
- **Per-member menu switcher (item 10)** — mirror `shop-for-picker.tsx` on the menu screen: a member/household toggle on `apps/web/app/(app)/menu/_components/menu-view.tsx` that filters rendered slots by `target_member_id` (household = all). URL-param driven like `?shop_for=`. Pure read-side filter; new small component `menu-member-picker.tsx` under the menu `_components/`.

## Verification
- **Gate:** the **Phase 8 design review** (design-system-architect + ux-reviewer + accessibility-auditor) must be signed off before this block runs — final verification is the last step, after design sign-off.
- **Determinism regression (must hold) — and it's cheap because v2.0 never touches the engine:**
  - `accepted_seed` byte-identical before/after any cook-status / inventory / shopping / **ingredient-substitution (Phase 6)** write.
  - `recomputeGroceryListsForMenu` output unchanged by inventory state; **changes correctly and only** when a Phase 6 ingredient override exists for the menu.
  - Engine input builder selects no execution/inventory/override columns.
  - **No golden-snapshot regen** — the engine is untouched. (Snapshots regen in **[v2.1](./v2.1.md)**.)
  - Cover the substitution invariants with the **constraint-menu-generator-life-cycle-test** skill.
- **Integration (real local Supabase, RLS on):** role matrix + soft-delete visibility for every new table (`inventory_items`, `shopping_sessions`/`shopping_item_status`, `slot_completions`, `menu_slot_ingredient_overrides`); shopping completeness thresholds; **per-leftover** auto-expire (each row expires on its own `expiration_date`, default from `max_storage_days` else `leftover_max_days`); inventory annotation shows required + on-hand + suggested-to-buy, never lowers `required`, never writes `grocery_items`; **all-members aggregation** sums shared + per-member correctly; **ingredient-substitution** validation rejects an allergen-introducing substitute (422). Author with **vitest-integration-author**.
- **Manual E2E:** `pnpm dev`, accept a menu → open shopping session (grouped by food group) → mark partial → see derived alert on affected recipe → mark a slot cooked → leftover appears with its own per-item expiry (editable) → next week's grocery view shows each line as "required (you have N · buy M)" → toggle the grocery "Everyone" view to see consolidated totals → switch the menu view between members → substitute green→red tomato on one slot and confirm the grocery list updates while the menu's seed/identity does not.
- Gates: `pnpm typecheck && pnpm test`, then `INTEGRATION_ENABLED=1 … pnpm test`, then `node scripts/verify-flow.mjs`.
