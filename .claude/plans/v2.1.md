# v2.1 — Smarter generation + Addons (the engine work + accompaniments)

> Part of the **[v2 epic](./v2.md)**. Sibling releases: **[v2.0 — Execution & Pantry](./v2.0.md)**,
> **[v2.2 — Extras](./v2.2.md)**, **[v2.3 — demo-lab + Bruno tooling](./v2.3.md)**, and **[v2.4 — Hosting](./v2.4.md)**. The community track moved to its own
> epic, **[v4](./v4.md)**. Read the [epic index](./v2.md) first for shared context, the locked decisions,
> and **the one hard determinism rule**. This file is the buildable plan for v2.1.

## What v2.1 is

Three tracks, run in parallel:

1. **The only engine work in the whole v2 epic** — item **11** (inclusive preferences + per-generation
   constraint overrides) and item **12** (multi-timeframe recipes). Both touch the engine *only at
   generation time* and both require an **intentional golden-snapshot regen**. Batching them means
   **one** regen against a stable engine, not two interleaved regens. *(Gated piece.)*
2. **The addons track (item 13)** — a new recipe *kind* (salsa / guacamole / dessert) that accompanies a
   meal but is **not** a constraint-engine candidate, plus **on-the-fly cook mode**. Co-located here
   because `recipe_kind` shares the recipe-classification surface with item 12's `recipe_meal_types` — but
   **it is engine-free**: addons are excluded at the *input-builder* boundary, never enter a snapshot, and
   **do not gate on the determinism regen**. Fully parallel to the engine work.
3. **The bulk recipe-create primitive (Track E)** — a transactional "create N recipes (+ children) in one
   call" route + module. Small, engine-free, independently useful — and the **materialization primitive
   that de-risks [v3](./v3.md)** (AI import generates recipes through it) and **[v4.0](./v4.0.md)**
   (community deep-copy import is "bulk-create + provenance"). Pulled into v2.1 so both downstream features
   call it rather than each inventing its own insert path.

**Not in v2.1:** community recipes (now their own epic, **[v4](./v4.md)**: [v4.0](./v4.0.md) share &
duplicate → [v4.1](./v4.1.md) reviews → [v4.2](./v4.2.md) page) and the versioned seed-pack manifest (rides
with the community official pack in [v4.0](./v4.0.md)). v2.0's inline ingredient array + `food_group` is
sufficient until then.

Prereqs from **[v2.0](./v2.0.md)**: none structurally — v2.1 can start as soon as the engine is stable.

**The determinism contract still holds**: *same inputs + same seed = same output*, post-accept mutation
still forbidden. The snapshot delta is the *intentional* product of new optional generation inputs.

### `docs/PRD/PRODUCT_PRD.md`
Add new top-level sections, all tagged **(v2.1)**:
- **§22 Inclusive vs exclusive dietary restrictions** — **exclusive** (hard, never violated — current behavior) vs **inclusive** (soft preference: bias toward a liked tag/ingredient, e.g. fish, without excluding others). A member profile holds both. **At generation time** the user can add inclusive preferences, add extra exclusive restrictions, and **temporarily relax** a profile exclusive restriction for that one generation. All overrides are generation inputs reflected in `inputs_hash`; determinism preserved.
- **§23 Multi-timeframe recipes** — a recipe declares the **set** of meal timeframes it can fill (sandwich = breakfast + snack + dinner) instead of one. The recipe form's single meal-type Select becomes a multi-select; ≥1 required. The engine treats the recipe as a candidate for any slot whose meal matches *any* of its timeframes. Existing recipes keep their current meal as a one-element set (no behavior change until broadened). Substitution/cook-status/inventory unaffected.
- **§24 Addons & on-the-fly cook mode (item 13)** — an **addon** is a recipe whose *kind* is `addon` (salsa, guacamole, dessert): a real recipe with the same structure and cook-mode, but it is **never a constraint-engine candidate** — it accompanies meals, it doesn't fill a slot. Addons have **no meal-type requirement** (the ≥1 rule from §23 applies only to `kind='meal'`). A user can **attach an addon to an accepted menu** either week-wide or to a specific slot ("guac with Tuesday lunch"); attaching/detaching never changes the menu's identity or seed. Attached addon ingredients appear in the grocery list in a dedicated **"Addons" section** (separate from the meal-derived lines). **On-the-fly cook mode**: the user can open cook mode for *any* recipe — meal or addon — standalone, without it being on the active menu and **without affecting** it; the flow is ephemeral (optional leftover emission into inventory, reusing §16). Note explicitly: addons are excluded from generation at the input-builder boundary, so they're invisible to `accepted_seed` and the engine.
- §3 Recipe Management: note a recipe can be eligible for **multiple meal timeframes**, and that a recipe carries a **kind** (`meal` | `addon`) where addons are excluded from menu generation. Note recipes can also be created in **bulk** (the v2.1 primitive: many recipes saved in one transaction). *(Publish/import is documented in the community epic, [v4.0](./v4.0.md).)*
- **§25 Bulk recipe creation (v2.1)** — a single operation that creates **N recipes plus their ingredients / instructions / dietary tags / meal-types in one transaction**, returning the created recipe ids, reusing the same per-recipe validation as the single-create form. Engine-invisible (new recipes are ordinary catalog rows). It is the substrate the [v3](./v3.md) AI import (generate-from-list) and the [v4.0](./v4.0.md) community deep-copy import both build on; documented here so the contract is owned by the release that ships it.
- §11 label suggestions: note inclusive-preference tags draw from the same dietary-tag/ingredient label sets as exclusive restrictions.

### Track C — Inclusive preferences + generation-time overrides (item 11) — ENGINE WORK
Sequence schema/UI independently; the engine change is the gated piece (determinism snapshots).
- **Storage** — add inclusive prefs distinct from hard `member_dietary_restrictions` / `member_allergies`:
  `member_dietary_preferences` (`id, member_id, kind enum {dietary_tag,ingredient}, value text|ingredient_id, ts`; one row per liked tag/ingredient). (Alt: a `preference_mode` enum on a unified table — chosen table form keeps the existing hard tables untouched.) RLS mirrors the existing member-constraint sub-tables.
  Endpoints mirror `POST .../members/[memberId]/dietary-restrictions`: `POST|DELETE .../members/[memberId]/dietary-preferences`.
- **Engine input** (`packages/constraint-engine/src/types.ts`) — extend `MemberSnapshot` with `dietaryPreferences: { tags: string[]; ingredients: string[] }` (inclusive). Extend `GenerateMenuOptions` with `additionalDietaryPreferences?` (inclusive add at generation) and `relaxedDietaryRestrictions?` / `relaxedAllergies?` (subtractive — temporarily lift a profile *exclusive* restriction for this generation).
- **Engine logic** — the soft-bias is the first real soft-constraint use of the deferred scoring hook in `packages/constraint-engine/src/assign.ts`. Among the **hard-valid** candidates for a slot, partition into "preferred" (matches an inclusive tag/ingredient) vs "rest"; the seeded RNG picks from "preferred" when non-empty, else "rest". Keeps generation **fully deterministic** and never excludes a valid recipe. `filter.ts` is unchanged for inclusive prefs (never hard filters); `filter.ts` honors `relaxed*` by removing those values from the hard sets *before* filtering.
- **Overlay** (`apps/web/lib/api/menu-overlay.ts` `computeEffectiveOverlay`, `menu-input-builder.ts`) — extend the additive dedup to carry inclusive prefs, and add the subtractive `relaxed*` handling (remove from the member's effective hard set for this generation only). All into `inputs_hash` — a relaxed/preference-tuned generation is legitimately a *different* generation.
- **UI** *(promote from the approved design-lab mock — see Part 2.0 gate)* — surface inclusive prefs on the member profile editor, and a per-generation override panel on the menu-generation form (add inclusive prefs, add/relax exclusive restrictions for this run). Reuse existing constraint pickers.

### Phase 8 — Multi-timeframe recipes (item 12) — Track C / engine
The **second** engine-touching change. Generation-time only; preserves the determinism contract.
- **Schema** — replace scalar `recipes.meal_type` with junction **`recipe_meal_types (recipe_id uuid FK→recipes, meal_type public.meal_type, PK(recipe_id, meal_type))`**, mirroring `recipe_dietary_tags`; RLS via recipe ownership (reuse the recipe sub-table policy template). **Backfill**: one row per existing `recipes.meal_type`, then drop the scalar column. Enforce ≥1 meal type per recipe **(for `recipe_kind='meal'`)** at the write/route layer.
- **Types** — `packages/supabase/src/types/db.ts` `RecipeRow.meal_type: MealType` → fetch the set; `packages/constraint-engine/src/types.ts` `RecipeSnapshot.mealType: MealType` → `mealTypes: MealType[]`.
- **Engine** — `packages/constraint-engine/src/filter.ts` `isRecipeValidForSlot`: `recipe.mealType !== slot.mealType` → `!recipe.mealTypes.includes(slot.mealType)`. The **only** engine logic change; slot enumeration (`slots.ts`, driven by member `meal_frequency`) is **untouched**. Update `__tests__/filter.test.ts` meal-match cases.
- **Module / route / input builder** — `packages/supabase/src/module/recipes.ts` (RECIPE_SELECT + payload types accept `meal_types: MealType[]`), the recipes POST/PATCH handler, and the engine input builder all carry the set. A recipe broadening its meal set legitimately changes `inputs_hash`.
- **UI** *(promote from the approved design-lab mock — see Part 2.0 gate)* — `apps/web/app/(app)/recipes/_components/recipe-form.tsx`: the single meal-type Select becomes a multi-select (checkbox group or `MultiLabelCombobox`-style), default = the recipe's current set, ≥1 required.

### Track D — Addons + on-the-fly cook mode (item 13) — ENGINE-FREE, fully parallel
No snapshot regen; does not gate on Track C. Reuses the v2.0 Phase-6 menu-state pattern and the v1.9 Cook Sheet.
- **Recipe kind** — `recipes.recipe_kind enum {meal,addon} NOT NULL DEFAULT 'meal'` (`supabase-add-column`). Backfill all existing rows to `meal`. The §23 "≥1 meal type" rule is enforced **only for `kind='meal'`**; addons may have zero `recipe_meal_types`.
- **Engine exclusion (the safety boundary)** — `apps/web/lib/api/menu-input-builder.ts` filters its recipe query to `recipe_kind='meal'`. Addons never enter the `RecipeSnapshot[]`, never touch `inputs_hash`, never appear in a golden snapshot. **This is an input-selection change, not an engine code change** — `packages/constraint-engine/*` is untouched by Track D.
- **Menu attachment** — `menu_addons` (`id, menu_id, workspace_id, addon_recipe_id, target_slot_id NULL, servings numeric NULL, note text NULL, created_by, ts`; index `(menu_id)`). `target_slot_id NULL` = whole-week addon; set = tied to one slot. RLS via `menus.workspace_id` EXISTS, write = creator/admin. Keyed by `menu_id` → structurally invisible to `accepted_seed` (which hashes only meal-slot recipe-tuples), same proof as v2.0 Phase 6.
- **Grocery sourcing** — `grocery_items.source enum {meal,addon,extra} NOT NULL DEFAULT 'meal'` (`supabase-add-column`; `extra` dormant until [v2.2](./v2.2.md)). `recomputeGroceryListsForMenu` gains an addon pass: after meal aggregation, load `menu_addons` → their `recipe_ingredients` → emit/merge grocery lines tagged `source='addon'`. Still engine-agnostic (addons are menu state). The grocery UI groups by `source` into a dedicated **"Addons"** section.
- **On-the-fly cook mode** — add a recipe-initiated entry point to the v1.9 Cook Sheet that opens for any recipe (meal or addon) with **no menu write** — ephemeral checklist, optional "save leftover" that emits `inventory_items(source='leftover')` (reuses v2.0 §16). Distinct from v1.9 menu-slot cook mode, which writes `menu_slots.cooked_at`.
- **Modules / endpoints** — `menu-addons.ts`/`.react.ts`; `recipe_kind` carried through `recipes.ts` RECIPE_SELECT + payloads. Endpoints: `POST|DELETE .../menus/[menuId]/addons`, `GET .../recipes?kind=addon` (addon picker). After an addon write on an *accepted* menu, re-run `recomputeGroceryListsForMenu`.
- **UI** *(promote from the approved design-lab mock — see Part 2.0 gate)* — addon picker + attach control on the menu view (reuse the recipe picker); "Addons" grocery section; a "Cook now" affordance on recipe detail for on-the-fly mode; recipe form gains a `kind` toggle (meal/addon) that hides the meal-type multi-select when `addon`.

### Track E — Bulk recipe-create primitive — ENGINE-FREE, fully parallel
No schema, no snapshot regen; does not gate on Track C. The smallest of the three tracks.
- **Route / server action** — `POST .../workspaces/[id]/recipes/bulk` accepting `{ recipes: RecipePayload[] }`,
  each element validated by the **same Zod schema** the single-create form uses. Wrap the inserts in **one
  transaction** (recipe + `recipe_ingredients` + instructions + `recipe_dietary_tags` + `recipe_meal_types`
  [Phase 8] + `recipe_kind`): a single invalid payload rolls the **whole batch** back. Return the created
  recipe ids. Delegate to **route-handler-engineer**.
- **Module** — extend `packages/supabase/src/module/recipes.ts` with a `createRecipesBulk` (RO-RO, array in
  → ids out) reusing the existing single-create child-row logic; export from the barrel + `.react.ts` hook.
  Delegate to **supabase-module-author**.
- **Engine-free by construction** — new recipes are ordinary catalog rows; nothing here touches
  `inputs_hash`, the engine, or a snapshot. This is why Track E needs no regen and runs fully parallel.
- **Atomicity note (build status):** the initial `createRecipesBulk` implementation validates the whole
  batch *before* any write and uses a compensating `DELETE … IN (insertedIds)` (with `ON DELETE CASCADE`)
  if a child insert fails — so the spec'd guarantee "one invalid payload inserts nothing" holds, but a
  process crash in the narrow window between a child-insert failure and the compensating delete could
  orphan recipe rows. **Hardening before v3/v4 actually consume it:** promote to a true transactional
  `sys_create_recipes_bulk` Postgres RPC (`BEGIN/COMMIT`) authored via **supabase-migration-author**; the
  module then calls the RPC. Tracked here so it isn't lost.
- **Consumers (later releases, not built here)** — [v3](./v3.md) materializes AI-parsed recipes through it;
  [v4.0](./v4.0.md) layers lineage/provenance copy on top of it for community import.

### Track C — Determinism (the gated piece — both engine items share ONE regen)
Regenerate engine golden snapshots **once** with **determinism-snapshot-curator** after *both* item 11
and item 12 land. The snapshot delta is *intentional* (new optional input fields + soft-bias selection +
set-membership meal check). Prove:
- With **no** preferences/overrides **and** every recipe a one-element meal set, output is **byte-identical** to the pre-change snapshots (proves both backfills are no-ops).
- With inclusive prefs and/or `relaxed*` set, output is stable across runs for a fixed seed and never includes a hard-excluded recipe.
- A recipe broadened to multiple meals becomes a candidate for each matching slot; the delta is intentional and scoped. Slot enumeration counts are unchanged.

## Verification
- **Determinism regression (must hold) — ONE intentional snapshot regen:**
  - **Track C:** with no preferences/overrides, engine output is byte-identical to the pre-change golden snapshots; with inclusive prefs and/or `relaxed*` set, output is deterministic for a fixed seed and never includes a hard-excluded recipe.
  - **Phase 8:** a recipe whose meal set has exactly one element produces byte-identical output to the pre-change snapshots (proves the backfill is a no-op); a recipe broadened to multiple meals becomes a candidate for each matching slot and the snapshot delta is intentional and scoped. Slot enumeration counts unchanged.
  - Regenerate snapshots via **determinism-snapshot-curator** (intentional delta). Cover with the **constraint-menu-generator-life-cycle-test** skill.
- **Addons (Track D, engine-free — verify the boundary holds):**
  - Engine input builder selects **no** `recipe_kind='addon'` rows; an addon recipe **never** appears in a golden snapshot and **never** changes `inputs_hash`. (This is the proof Track D needs no regen.)
  - `accepted_seed` byte-identical before/after attaching/detaching a `menu_addons` row.
  - `recomputeGroceryListsForMenu` emits addon lines tagged `source='addon'`, meal lines `source='meal'`; the meal-line totals are **unchanged** by addon attachment.
- **Integration (real local Supabase, RLS on):** role matrix + soft-delete visibility for `member_dietary_preferences`, `recipe_meal_types`, `menu_addons`; backfill leaves every existing recipe with exactly one meal type and `recipe_kind='meal'`; RLS on `menu_addons` via `menus.workspace_id`; on-the-fly cook mode writes no `menu_slots` row. Author with **vitest-integration-author**.
- **Bulk-create (Track E, engine-free):** creating N valid recipes in one call inserts N recipes + all child rows and returns N ids; a batch containing **one** invalid payload inserts **nothing** (transaction rollback); bulk-created recipes are byte-identical in the engine to the same recipes created singly (no `inputs_hash`/snapshot effect). Author with **vitest-integration-author**.
- **Manual E2E:** generate a menu with an inclusive "fish" preference and a relaxed restriction and confirm it's deterministic → mark a sandwich recipe as breakfast + snack + dinner and confirm it can be assigned to any of those slots → create a `guacamole` addon recipe, attach it to Tuesday lunch, confirm it shows in the grocery "Addons" section while the menu seed/identity is unchanged → open "Cook now" on an unattached addon and confirm the active menu is untouched.
- Gates: `pnpm typecheck && pnpm test`, then `INTEGRATION_ENABLED=1 … pnpm test`.
