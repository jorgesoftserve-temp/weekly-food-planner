## What v2.2 is

The third step of the weekly-planner loop the user framed:

1. plan foods & menus for the week — *engine (v1) + smarter generation (v2.1)*
2. plan grocery shopping & pantry review — *v2.0 shopping/inventory*
3. **plan regular shopping & inventory for non-food pantry — _this release_**

**Extras** = grocery lines that aren't derived from the menu. "I ran out of milk for my smoothies, but
my protein smoothie isn't on the menu and I don't want to add it — I just want milk on the list." The
user can add arbitrary items (food or, eventually, non-food) to the week's grocery list in a dedicated
**"Extras"** section, completing the planner so it covers everything you'd actually buy.

## Why it's cheap after v2.1 (the deliberate seam)

v2.1's Addons track introduces `grocery_items.source enum {meal,addon,extra}`. The `extra` value is
**already reserved** there. So v2.2 is mostly:
- a manual "add to this week's list" UI that writes a `grocery_items` row with `source='extra'`;
- those rows render in the **"Extras"** grocery section (the sectioning machinery already exists for Addons);
- they are **pure grocery state** — never touch the menu, the seed, or the engine (same invariant class as Addons/substitution).

## Key open decisions (resolve before phasing)

- **Non-food items** — the user flagged "maybe non-food stuff, but that's something to think about."
  Recommended path: ship **free-text extras** first (an extra line may link a catalog `ingredient_id`
  *or* carry free text + unit, no catalog link required). A dedicated **non-food catalog** (with its own
  category/aisle taxonomy) is a later refinement, not a v2.2 blocker. This keeps step 3 usable without a
  catalog redesign.
- **Persistence scope** — are extras attached to a specific week's `menu`/grocery list, or to a standing
  workspace "shopping list" that persists across weeks? Likely per-week to match the existing grocery
  model, with a possible "recurring extras" convenience later.
- **Inventory interplay** — purchased extras should be eligible to flow into `inventory_items`
  (`source='purchase'`) on shopping finalize, exactly like meal-derived purchases (v2.0 §13). Confirm the
  finalize path treats `source='extra'` lines the same.

## Likely surface (to be detailed)
- Schema: reuse `grocery_items` + `source='extra'`; possibly `grocery_items.label`/free-text columns; an optional non-food flag on `ingredients` (deferred).
- `apps/web/lib/grocery-filter.ts` / grocery UI — render the "Extras" section; add-extra control.
- Finalize path (`shopping-finalize.ts`) — extras spill into inventory like any purchase.

## Carried-over follow-ups from v2.1

These were surfaced during the v2.1 close-out (2026-06-12) and deliberately **not** folded into the
v2.1 release — none blocked ship, and the parity auditor signed off "ship anyway." They are parked here
so v2.2's build picks them up. All are engine-free and snapshot-free.

### Cosmetic / parity nits (UI polish — defer to `ux-reviewer` + `design-parity-auditor` at build)
- **Radius deficits** — a few promoted surfaces use `rounded-xl` where the design-lab mock specifies
  `rounded-2xl` (e.g. `MemberOverridePanel`), and `InlineChipInput` renders `rounded-md` against a
  `rounded-lg` mock. Bring the live radii in line with the mocks.
- **Member-form hard-constraint container not promoted** — the member form's hard-constraint group still
  lacks the red-tint (`--destructive`-derived) container the design-lab mock shows; promote it so hard
  restrictions read as visually distinct from soft preferences.
- **Grocery legend radius** — minor radius mismatch on the grocery section legend chip vs. mock.

### Engine-free hardening (pick up alongside the v3/v4 bulk-import seam)
- **`createRecipesBulk` → transactional RPC** — the v2.1 bulk recipe-create primitive currently inserts
  recipe + children across multiple client calls (best-effort, not all-or-nothing). Before
  [v3](./v3.md) AI-import and the [v4](./v4.md) community deep-copy import consume it at volume, harden it
  into a single `sys_create_recipes_bulk` RPC (recipe + `recipe_meal_types` + ingredients + steps in one
  transaction, returning created ids). Delegate to `supabase-migration-author` (RPC + RLS in one file)
  then re-point `apps/web/lib/api/recipes-bulk.ts` at the RPC.

### Pre-existing, out of v2.1 scope (track, don't block)
- **`seed-ingredients` `sys_save_label` 500** — the admin seed-ingredients route returns a non-fatal
  "Authentication required" 500 from `sys_save_label`; the seed still succeeds for the catalog rows that
  matter. Investigate the auth context the RPC runs under (admin client vs. RLS-gated RPC). Non-blocking.
- **design-lab v2.0 mock `menu-exec-screen.tsx` `totalSlots` artifact** — a `totalSlots is not defined`
  runtime artifact lives in the **protected** v2.0 design-lab mock, not the live app (no live crash). Do
  **not** edit the mock until the v2 epic ships (see [[keep-design-lab-mocks-until-v2-epic-done]]); fold
  the fix into the eventual mock retirement.

## Determinism
Trivially safe — extras are pure grocery state. `accepted_seed` and the engine are untouched; no snapshot
regen. Same verification class as v2.0 Phase 6 / v2.1 Track D. The carried-over follow-ups above are
likewise engine-free and snapshot-free.
