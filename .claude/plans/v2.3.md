## What v2.3 is

The core community capability: **a user shares a workspace recipe to the community, and other users import
a deep-copy *duplicate* into their own workspace.** This is **copy semantics, not collaboration** — there is
**no shared mutable row**. Each import is an independent recipe the importer owns and can edit freely. This
slots cleanly under the existing workspace-scoped, single-owner RLS model (an **additive** published-read
policy; the existing `recipes_read` is never widened).

**Not in v2.3:** voting/reviews ([v2.4](./v2.4.md)) and the browse/discovery page + "My community" tab
([v2.5](./v2.5.md)). v2.3 ships publish + import + the provenance/ownership backbone, plus the **official
starter pack** (system-workspace recipes published `is_official=true`, imported through the same flow) and
the **versioned seed-pack manifest** that feeds it.

## The ownership / provenance model (the part to get right first)

The user's requirement: **track the original creator of a recipe even inside duplicates**, so that (a) no
one can re-upload the *same* recipe to the community twice, (b) authorship/"copyright" is attributable to
avoid disputes, and (c) this all holds **even if the original creator deleted their recipe and an importer
edits their copy into a variant**. Plus: surface **variants from other users** in the community.

Two immutable, copy-on-import columns on `recipes` carry the lineage. **Neither is a FK to `recipes`**, so
deleting the original never breaks them:

- **`lineage_id uuid NOT NULL`** — the stable identity of "this recipe and all its descendants." Generated
  at *original* creation; **copied verbatim** into every imported duplicate and preserved through variant
  edits. Groups originals + all variants/copies across all workspaces. Used for **anti-double-publish** and
  **variant grouping**. (A plain uuid, not a FK — survives deletion of the root recipe.)
- **`original_creator_id uuid`** (→ `auth.users`, `ON DELETE SET NULL`) + **`original_creator_name text`**
  (denormalized snapshot) — **immutable** attribution copied verbatim on import. Survives deletion of the
  source recipe *and* of the creator's account (the name snapshot persists). This is the "copyright"/credit
  field; an importer's edits never change it.

Existing provenance fields still track the *immediate* hop (distinct from the lineage *root*):
`source_publication_id`, `imported_from_recipe_id`, `imported_at`.

**Backfill** (one-time): every existing recipe becomes its own lineage root — `lineage_id = <fresh uuid>`,
`original_creator_id = recipes.created_by`, `original_creator_name = <snapshot>`.

### Rules the lineage enforces
1. **No double upload of the same recipe.** A publish is rejected if a *live, canonical* publication already
   exists for this `lineage_id`. Enforced structurally: **partial unique index on `recipe_publications(lineage_id) WHERE NOT is_unpublished AND NOT is_variant`**. You cannot list the same original twice.
2. **Importers can't claim authorship.** Because `original_creator_id` is copied and immutable, an imported
   copy always credits the original creator — an importer can never publish it *as their own original*. If
   an importer publishes their (edited) copy, it is necessarily a **variant** (rule 3), still crediting the
   original.
3. **Variants are allowed and grouped.** A publication whose `lineage_id` already has a canonical entry is
   flagged `is_variant=true`, attributed to `original_creator_id` **plus** `variant_author_id` (the editor).
   The community groups variants under their `lineage_id` (surfaced in [v2.5](./v2.5.md)). This holds **even
   if the original was deleted** — the lineage and creator snapshot live on the publication row, not the
   (possibly soft-deleted) source recipe.

### Owner deletes a shared recipe (the user's flow #4)
On soft-delete of a source recipe, the owner chooses whether to **also remove it from the community**. If so,
its publication is set `is_unpublished=true` (kept for rank/provenance history). **Existing imported copies
are untouched** — they simply **become ordinary non-shared recipes** in their workspaces (their `lineage_id`
+ `original_creator_id` persist for attribution, but there's no live community link). No cascade, no orphan.

> **Flagged sub-feature (candidate for its own slice):** "on update, see the *original* version and revert
> vs. keep my copy" (prior flow #5) needs **version snapshots** of the published recipe. The lineage model
> here is the prerequisite; the diff/revert UX is deferred — slot it after v2.3 once lineage is proven.

## Likely surface (to be detailed at phase time)
- **Schema** — `recipes`: `lineage_id`, `original_creator_id`, `original_creator_name`, `source_publication_id`, `imported_from_recipe_id`, `imported_at` (all via `supabase-add-column`; **display/provenance only, never engine input**). `recipe_publications` (`id, recipe_id, workspace_id, published_by, lineage_id, original_creator_id, original_creator_name, is_official, is_variant, variant_author_id NULL, is_unpublished, import_count, title_snapshot, cuisine_snapshot, ts`; partial unique `(lineage_id) WHERE NOT is_unpublished AND NOT is_variant`; index `(is_unpublished) , (is_official) WHERE is_official`, `(lineage_id)` for variant grouping). *(Upvotes/reviews columns are added in [v2.4](./v2.4.md).)*
- **RLS** — additive `recipes_read_published` (+ mirror on recipe sub-tables) `USING (is_deleted=false AND EXISTS(live publication))` — do **not** widen `recipes_read`. Publications: read `USING(NOT is_unpublished)` for authenticated + owner sees own; write = owning-workspace creator/admin. (Template: `migrations/20260523000602_rls_create_recipe_policies.sql`.)
- **Import** — deep copy via `apps/web/lib/api/recipe-import.ts` reusing `createRecipe` (copies recipe + ingredients + instructions + dietary_tags + **`recipe_meal_types`** [v2.1] + **`recipe_kind`**; **copies `lineage_id` + `original_creator_id` + `original_creator_name` verbatim**; sets `source_publication_id`/`imported_from_recipe_id`/`imported_at`; labels funnel through `sys_save_label`; global `ingredient_id` FKs need no copy); bump `import_count`.
- **Publish/unpublish/variant** — route validates the lineage rules (1–3 above) at the handler layer; official starter pack publishes from a **system workspace** with `is_official=true` through the same path.
- **Seed-pack** — move ingredient + starter-recipe seed out of the inline 30-item array into a **versioned, committed manifest** `packages/supabase/seed-pack/{ingredients,recipes}.json` (Zod-validated, offline-sourced & reviewed). Starter recipes seed into the system workspace then publish `is_official` — "seed for new users" and "official pack" become one mechanism. Idempotency: ingredients upsert on `name`; recipes keyed by stable `slug`/`external_key`.
- **Modules / endpoints** — `recipe-publications.ts`/`.react.ts`, `recipe-import.ts`. `GET /api/community/recipes` + `[publicationId]` (basic; full browse UI is [v2.5](./v2.5.md)); `POST .../workspaces/[id]/recipes/[recipeId]/publish`, `DELETE` (unpublish), `POST .../workspaces/[id]/recipes/import`.
- **UI** — publish action + lineage/credit badge on recipe detail; minimal import entry point (the rich browse/filter surface is [v2.5](./v2.5.md)).

## Determinism
Engine-free. `lineage_id`/`original_creator_id`/provenance are **never engine input** (same rule as all
provenance fields). An imported recipe yields the same menu as its source under identical constraints —
verify this in the lifecycle test. No snapshot regen.

## Verification (highlights)
- **Anti-double-publish:** publishing a second canonical entry for an existing `lineage_id` is rejected (partial-unique / 409).
- **Attribution survives deletion:** delete the original recipe + unpublish → an importer's copy still shows `original_creator_name`; the importer publishing an edited copy creates a **variant** crediting the original.
- **Import determinism:** an imported recipe generates the same menu as its source under identical constraints.
- **Cross-workspace RLS role matrix:** publisher in workspace A, importer in workspace B — B can browse/preview/import but cannot edit/unpublish A's publication; one workspace's `recipes_read` is never widened. *(Author with **vitest-integration-author**; this is new cross-workspace territory — see the agentic readiness notes.)*
