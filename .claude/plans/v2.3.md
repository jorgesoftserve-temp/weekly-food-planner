# v2.3 — demo-lab harness + Bruno collection tooling

> Part of the **[v2 epic](./v2.md)**. A developer-experience + go-to-market release: a **production-gated
> walkthrough harness** to demo the whole product to prospective users, and a **kept-current API test
> collection** (Bruno) with a one-call dev-onboarding seed.
> Read the [epic index](./v2.md) first for shared context and the determinism rule.
>
> **Status: planned.** Slots between [v2.2](./v2.2.md) (Extras) and [v2.4](./v2.4.md) (Hosting &
> Deployment). Hosting moved to [v2.4](./v2.4.md) to make room for this release.

## Why this release

Two gaps surface once v2's execution/pantry surface exists and we start thinking about beta:

1. **There is no way to *show* the product to a prospect** without standing up a full account, seeding
   data, and clicking through the real app — fragile for a sales/demo context. We already have the
   [`design-lab`](../../apps/web/app/\(design\)/design-lab/) pattern (a mock harness with a viewport
   toggle and Figma-prototype-style navigation); a sibling harness aimed at a **guided end-to-end story**
   is the right tool.
2. **The Bruno API collection has drifted.** `scripts/weekly-food-planner-bruno.json` predates v2.0 and is
   missing ~20 routes (inventory, leftovers, cook/completion, shopping-sessions, draft/accept/preview/
   history, alerts, ingredient-overrides, usability, constraints). A new dev can't get a working tour
   without hand-entering ingredients and recipes first.

Neither touches the engine, schema, or any golden snapshot (see Determinism).

## Scope

### 1. `demo-lab` harness — guided product walkthrough

A new mock harness whose job is to **demo the entire system to potential users**: a guided story across
**login → members → recipes → menu → grocery → shopping**, mostly static with **minimal interaction**
(Figma-prototype-style navigation between screens). Distinct from `design-lab`, which hosts isolated
component mocks for design review.

- **Location:** new route group `apps/web/app/(demo)/demo-lab/` mirroring the
  [`(design)/design-lab/`](../../apps/web/app/\(design\)/design-lab/) structure — `page.tsx` (control bar +
  canvas), `layout.tsx` (gating), `frame/page.tsx` (width-locked iframe for device viewports), and
  `_components/{screens.tsx, screens/*, mock-data.ts, cozy-shell.tsx}`.
- **Reuse, don't reinvent:** the design-lab **viewport segmented control**, the **width-locked `frame`
  iframe** (so responsive breakpoints fire honestly), the **`LabNavProvider`** click-to-navigate prototype
  pattern, and the **cozy mock toolkit** (`mock-image`, `recipe-icon`). The difference is the *content
  model*: demo-lab screens are ordered into a narrative the prospect walks through, not a flat registry of
  component states.
- **Production-visible, gated.** Unlike design-lab (`notFound()` when `NODE_ENV === 'production'`),
  demo-lab's `layout.tsx` gates on an env flag: render when `process.env.ENABLE_DEMO_LAB` is set, else
  `notFound()`. This lets a prospect be shown the **live hosted URL** once [v2.4](./v2.4.md) ships (the
  flag is wired into the v2.4 env matrix). Off by default, so a normal production build does not expose it.
- **Version banner.** A top bar reads **"Representing vX.Y"**, sourced from a single
  `DEMO_LAB_VERSION` constant. X.Y is the **epic's last shipped version** — e.g. an epic that ended at
  v2.2 shows "Representing v2.2". The refresh skill (below) bumps this constant.
- **Refresh cadence: epic-wrap / on-demand only — never per-minor.** At epic-wrap the shipped
  functionality equals what the demo should portray, so that is the natural sync point; ad-hoc refreshes
  happen only on explicit request. Between those points the harness is intentionally allowed to lag.

### 2. demo-lab maintainer agents/skills

Specified here, **authored during the v2.3 build** (they operate on the harness, which does not exist
yet). They mirror the existing design-lab tooling triad (`promote-design-lab-mock` skill +
`design-parity-auditor` agent + `design-lab-parity-check` skill):

- **Skill `refresh-demo-lab`** — trigger: **epic-wrap or explicit ask**. Walks each live view, regenerates
  or updates the matching demo-lab mock to reflect current functionality, and **bumps `DEMO_LAB_VERSION`**
  to the epic's last shipped version. Produces a short diff summary of what changed per screen.
- **Agent `demo-lab-curator`** (lean, optional) — owns the harness end-to-end for larger refreshes that
  span many screens; read-mostly and build-capable for the mock files, but **defers visual tokens to
  `design-system-architect`** and never edits live-app code.

### 3. Bruno collection currency + walkthrough seed

- **Walkthrough-seed endpoint `/api/admin/seed-walkthrough`** — a new admin-key-gated route under
  `apps/web/app/api/admin/`, matching the existing
  [`seed-ingredients`](../../apps/web/app/api/admin/seed-ingredients/route.ts) /
  [`seed-recipes`](../../apps/web/app/api/admin/seed-recipes/route.ts) pattern. **Idempotent.** In one call
  it seeds a fresh workspace with the **ingredient catalog + starter recipes + a sample member**, so a new
  dev (or a Bruno run) has a complete, *generatable* tour without hand-entering anything. It composes the
  existing seed helpers; it does **not** invent new persistence. Engine-invisible — ordinary catalog/recipe
  rows via existing validated paths.
- **Bruno collection refresh** — bring `scripts/weekly-food-planner-bruno.json` current with the full
  `apps/web/app/api/**` surface (the ~20 missing v2.0 routes), and add a **"0. Walkthrough setup"** folder
  that calls `seed-walkthrough` so the collection is runnable top-to-bottom from a clean stack. Preserve
  the existing folder order, collection variables, and `pm.test` scripts.
- **Skill `bruno-collection-sync`** — trigger: **end of every minor version**. Diffs `apps/web/app/api/**`
  against the collection, adds/updates requests (collection variables, `pm.test` scripts, and folder
  ordering preserved), and reports drift. An **agent is not warranted** — this is a deterministic
  diff-and-fill checklist, the same grain as the existing
  [`add-route-handler`](../../.claude/skills/add-route-handler/) skill.

## Out of scope
- Hosting / actually deploying demo-lab to a public URL — that is [v2.4](./v2.4.md); v2.3 only builds the
  harness and the `ENABLE_DEMO_LAB` gate it reads.
- Any schema change. `seed-walkthrough` writes ordinary rows through existing validated paths; no new table.
- Touching the live app's screens — demo-lab is a separate route group with its own mock data.

## Determinism
Engine-free, schema-free, snapshot-free. demo-lab is static mocks; the Bruno collection is a dev test
harness; `seed-walkthrough` inserts ordinary catalog/recipe rows exactly like the existing `seed-recipes`
admin route — it never feeds the engine and never touches `accepted_seed`. **No regen.**

## Verification
- **demo-lab renders** every screen across the device viewports in light + dark, navigates as a prototype,
  shows the correct **"Representing vX.Y"** banner, and is reachable **only** when `ENABLE_DEMO_LAB` is set
  (404s otherwise, including a default production build).
- **`seed-walkthrough` is idempotent** — running it twice leaves the workspace unchanged, and the seeded
  workspace **generates a menu** end-to-end (proves catalog + recipes + member are coherent).
- **Bruno collection runs green** top-to-bottom against a freshly seeded local stack (start with
  "0. Walkthrough setup"); every `apps/web/app/api/**` route has a corresponding request.
- **Skills behave on a dry run** — `bruno-collection-sync` reports zero drift immediately after the
  refresh; `refresh-demo-lab` bumps `DEMO_LAB_VERSION` and emits a per-screen change summary.
