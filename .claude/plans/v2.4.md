## What v2.4 is

The feedback layer on top of v2.3's publications: users **vote** on shared recipes and leave a lightweight
**review + comment** (not just a score). These produce the **ranking signals** the community page
([v2.5](./v2.5.md)) sorts by.

The user's flow items covered here: **#2** (a recipe gets upvoted/downvoted and reviewed) and **#6** (a
review system with small reviews + comments, not only voting).

## Scope
- **Voting** — `recipe_upvotes` (`PK(publication_id, user_id)` → structural one-vote-per-user; trigger
  maintains `recipe_publications.upvote_count`). The user described **up- *and* down-voting**, so the row
  carries a `value smallint CHECK (value IN (-1, 1))` (or split counts); the maintained aggregate is a net
  score plus raw up/down for ranking. Changing your vote updates in place (PK dedup); removing it deletes the row.
- **Reviews & comments** — `recipe_reviews` (`id, publication_id, user_id, rating smallint NULL, body text, ts`;
  one review per user per publication — `UNIQUE(publication_id, user_id)`; editable/deletable by author).
  Reviews attach to the **publication** (the shared thing), and via its `lineage_id` they aggregate across
  the lineage if desired.
- **Aggregates for ranking** — denormalize the signals v2.5 sorts by onto `recipe_publications`
  (`upvote_count`, `downvote_count` or net `score`, `review_count`, `avg_rating`), maintained by triggers.
  `import_count` already lands in v2.3.
- **Moderation** — noted as a **future surface** (a moderation/report queue is out of scope for v2.4).

## Likely surface (to be detailed)
- Schema: `recipe_upvotes`, `recipe_reviews`; aggregate columns + triggers on `recipe_publications`. RLS: vote/review write `WITH CHECK(user_id = auth.uid())`; read follows the published-read policy from v2.3.
- Modules: `recipe-upvotes.ts`/`.react.ts`, `recipe-reviews.ts`/`.react.ts`.
- Endpoints: `POST|DELETE .../community/recipes/[publicationId]/upvote`, `POST|PATCH|DELETE .../community/recipes/[publicationId]/reviews`.
- UI: vote control + review/comment thread on the community recipe detail (the detail view from v2.3, enriched).

## Determinism & RLS
Engine-free; votes/reviews are pure community state, never engine input, no snapshot regen. **One vote per
user / one review per user** is enforced structurally (PK / unique). Cross-workspace role-matrix tests
(a user in any workspace can vote/review any live publication but not edit others' votes/reviews) —
**vitest-integration-author**.
