## What v2.4 is

The repo is **local-only** today: no hosted-Supabase env, no production `.env`, no deploy step in CI
(`.github/workflows/ci.yml` runs typecheck + test only). Nothing for real users matters until there's a
hosted target. v2.4 stands that up — managed Supabase + Vercel + a deploy/seed CI step **plus an
agentic release pipeline** — so the execution/pantry features (v2.0–2.2), the [demo-lab](./v2.3.md), and
the AI import (v3) are something beta users can actually open in a browser. This is **ops/config work,
not product features and not AI**; no per-request cost.

## Scope

- **Managed Supabase project** — provision; push the existing migration set with `supabase db push`
  (migrations are already the source of truth — no schema authoring here). Wire `SUPABASE_PROJECT_REF` +
  access token for the `supabase-remote` MCP server (already configured in [`.mcp.json`](../../.mcp.json),
  currently unused).
- **Env layout** — production `apps/web/.env.production` template; the service-role key stays server-only
  (the **three-clients rule** is unchanged: `supabaseClient` / `supabaseServerClient` / `supabaseAdminClient`).
  Document secrets handling (Vercel project env + Supabase keys; never in the repo).
- **Seed-on-deploy** — run the admin seed route (or a migration-embedded `INSERT … ON CONFLICT` of the seed
  manifest) **once, idempotently**, against the hosted DB as a post-deploy step. (The versioned seed-pack
  manifest itself lands with the community official pack in [v4.0](./v4.0.md); until then v2.0's inline
  30-item seed is what gets pushed. The [v2.3](./v2.3.md) `seed-walkthrough` route is the dev-onboarding
  analogue — same admin-key pattern, never run against production.)
- **Deploy job in CI** — Vercel for [`apps/web`](../../apps/web) + a migrate/seed step. Absent today.
- **Already designed deploy-aware** in v2 (e.g. `ADMIN_API_KEY` required for the admin/seed surface, a
  stable system-workspace id for the official pack) — v2.4 wires those to real production values.

### Deployment config plan (Vercel + Supabase Cloud)

The concrete configuration the above scope resolves to. All secrets live in the platform's env store
(Vercel project env, Supabase dashboard) — **never** committed.

| Variable | Where | Notes |
|---|---|---|
| `NEXT_PUBLIC_SUPABASE_URL` | Vercel (public) | hosted project URL |
| `NEXT_PUBLIC_SUPABASE_ANON_KEY` | Vercel (public) | anon/publishable key — browser-safe |
| `SUPABASE_SERVICE_ROLE_KEY` | Vercel (server-only) | powers `supabaseAdminClient`; never in the client bundle |
| `SUPABASE_PROJECT_REF` | CI / local | targets `supabase db push` + the `supabase-remote` MCP |
| `SUPABASE_ACCESS_TOKEN` | CI / local (secret) | CLI + `supabase-remote` MCP auth |
| `ADMIN_API_KEY` | Vercel (server-only) | gates the `/api/admin/*` seed surface |
| `ENABLE_DEMO_LAB` | Vercel (per-deployment) | turns on the [v2.3 demo-lab](./v2.3.md) in a hosted build for sales/demo |
| `ANTHROPIC_API_KEY` | Vercel (server-only) | v2.0 food-group classifier; reused by [v3](./v3.md) |

- **Three-clients discipline holds in production:** only `NEXT_PUBLIC_*` reaches the browser; the
  service-role key and `ADMIN_API_KEY` are server-only Vercel env. CI introspection (`supabase-remote`)
  confirms RLS is enabled on every table.
- **Branch model:** `stage` is the integration branch; a dedicated **deploy branch** (or a tagged release
  off `stage`) triggers the production pipeline. Vercel preview deployments cover PRs.

### Release-management agents/skills (authored in the v2.4 build)

The user wants releases — GitHub **and** cloud, for both the DB and the rest of the code — handled
agentically as a repeatable pipeline. These are **specified here, authored during the v2.4 build** (they
reference hosted infrastructure that does not exist until v2.4 provisions it).

- **Agent `release-manager`** — owns cutting a release end-to-end across GitHub **and** cloud. Orchestrates
  the pipeline below, decides go/no-go from the CI + smoke signals, and knows the documented rollback
  path. Sequences the skills; does not duplicate their logic.
- **Skills (discrete, individually re-runnable steps `release-manager` calls):**
  - **`deploy-supabase-migrations`** — push the migration set to the cloud project via the `supabase-remote`
    MCP / `supabase db push`; verify `list_migrations` on the remote matches the local set before
    proceeding.
  - **`deploy-vercel`** — promote `apps/web` to the production Vercel deployment; confirm the build and the
    env matrix.
  - **`cut-github-release`** — tag the release, append the changelog entry (mirrors
    [`docs/agentic/changelog/`](../../docs/agentic/changelog/)), and create the GitHub release.

### Deployment pipeline

```
push to deploy branch
  → CI: typecheck + test            (a red test blocks everything below)
  → deploy-supabase-migrations      (db push to cloud; list_migrations parity check)
  → deploy-vercel                   (promote apps/web)
  → hosted smoke (verify-flow.mjs against the Vercel URL)
  → cut-github-release              (tag + changelog + GH release)
  → [on failure] documented rollback path
```

## Out of scope
- Any schema change (migrations already exist; v2.4 only *applies* them to a hosted project).
- The native mobile app — that is its own version, **[v5](./v5.md)**. The hosted web app already serves
  mobile/tablet viewports responsively, which is what beta users get.
- AI features (v3) — v2.4 only makes the *place* for them to run.
- Building the [demo-lab](./v2.3.md) or the Bruno tooling — that is v2.3; v2.4 only wires the
  `ENABLE_DEMO_LAB` flag into the hosted env so demo-lab can be shown on the live URL.

## Determinism
Engine-free, schema-free. Hosting changes *where* the code runs, not *what* it computes — the determinism
contract and golden snapshots are untouched. No regen.

## Verification
- **Migrations apply cleanly** to the managed project (`supabase db push` from a clean state); `list_migrations`
  via the `supabase-remote` MCP matches the local set.
- **Seed-on-deploy is idempotent** — running it twice leaves the catalog unchanged (upsert keys hold).
- **Hosted smoke** — the end-to-end flow ([`scripts/verify-flow.mjs`](../../scripts/verify-flow.mjs)) passes
  against the hosted target (point its base URL at the Vercel deployment).
- **Three-clients rule intact** — service-role key never reaches the browser bundle; `supabase-remote`
  introspection confirms RLS is enabled on every table in production.
- **CI deploy** — a push to the deploy branch runs typecheck + test, then migrate + deploy, then the smoke
  check; a red test blocks the deploy.
- **Release pipeline dry-run** — `release-manager` sequences the three skills against a staging/preview
  target; each skill is independently re-runnable and the rollback path is exercised once.
