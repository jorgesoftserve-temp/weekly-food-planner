# v2 Epic: Execution & Pantry + Community Recipes — PRD updates & implementation plan

> Roadmap: **[v1](./v1.md)** (MVP, shipped) → **[v1.5](./v1.5.md)** (enhancements + agentic/MCP tooling,
> shipped) → **v2** (this doc, planned) → **[v3](./v3.md)** (AI assist POC + hosted deployment, research).

## Context

The Weekly Food Planner today ends its lifecycle at **accept** : a menu is generated
deterministically, a grocery list is recomputed from it, and everything is read-only
afterward. The user wants to extend the product *past* accept — into the messy reality of
actually shopping and cooking the week:

1. **Inventory** with optional/manual expiration and partial-spoilage ("2 of my 5 tomatoes rotted").
2. **Grocery shopping confirmation** — review the whole list or grouped by food group (food group classified for ingredients).
3. **Incomplete-shopping alerts** — when only 30–90% of ingredients were found, surface which recipes of the week are affected.
4. **Cook-status on ongoing menus** — confirm which weekly slots were actually cooked vs skipped.
5. **Leftovers** — a leftovers inventory, manually marked eaten or auto-trashed after a configurable max (default 3 days).
6. **Community recipes** — publish/rank/import recipes cross-workspace so new users don't re-enter basics.

A second round of refinements (items 7–11) extends the same epic:

7. **Per-leftover spoilage length** — each leftover spoils on its own clock, not one global setting; every leftover row carries its own editable expiration, defaulted intelligently.
8. **All-members aggregated grocery view** — beyond the existing shared + per-member split, a single consolidated household list combining everything you'd actually buy.
9. **Menu-level recipe-ingredient substitution** — swap an ingredient *for this accepted menu only* (red tomato instead of green). The grocery list reflects the swap; the deterministic seed does **not** change.
10. **Member switcher on the menu view** — per-member view buttons on the menu screen, mirroring the grocery shop-for picker, so you can see each member's plan.
11. **Inclusive (preference) restrictions** — alongside hard **exclusive** restrictions (no peanuts), soft **inclusive** preferences (prefers fish, not required). Both can be tuned *per generation* (on a diet this week, off it next), without breaking determinism.

A third round (item 12) plus two cross-cutting refinements (a v2 seed-pack, and a v3 AI/deployment track) extend the epic further:

12. **Multi-timeframe recipes** — a recipe can be eligible for *multiple* meal timeframes (a sandwich works as breakfast, snack, *and* dinner), not the single `meal_type` it carries today.

Items 1–8 and 10 are **post-accept mutable state or pure read-side views**. The single hard rule
that shapes the whole design: **none of it may feed the constraint engine or change `accepted_seed`** —
the engine stays pure and deterministic (`apps/web/lib/api/menu-accept.ts` proves `accepted_seed`
derives only from slot recipe *tuples* `{day, meal_key, target_member_id, recipe_id}`;
`apps/web/lib/api/menu-grocery.ts` `recomputeGroceryListsForMenu` must stay inventory-agnostic).
Inventory only ever touches grocery on the **read side** as a non-destructive **annotation**
(show required + on-hand + suggested-to-buy), mirroring the existing presentation-only
`applyShopForFilter` in `apps/web/lib/grocery-filter.ts`.

**Deliberate exceptions to "don't touch the engine / seed", all safe:**
- **Item 9 (ingredient substitution)** is *menu state, not inventory*. Because `accepted_seed` hashes
  only slot recipe-tuples (never ingredient lists), a per-slot ingredient override changes the
  **grocery recompute output** but is **structurally invisible to `accepted_seed` and to the engine**.
  Recompute legitimately reflects it because the substitution is part of the menu the user accepted/edited.
- **Items 11 & 12 are the two engine-touching items**, and both touch the engine *only at generation time*.
  Generation inputs are *allowed* to change `inputs_hash` (a different week, or a recipe newly eligible for
  more meals, is a different generation). The determinism contract — *same inputs + same seed = same output* —
  is fully preserved, and **post-accept mutation is still forbidden**. Both require an intentional
  golden-snapshot regen.
  - **Item 11 (inclusive prefs + per-generation overrides)** adds a deterministic *soft-bias* to candidate
    selection and lets the per-generation overlay add/relax constraints.
  - **Item 12 (multi-timeframe recipes)** broadens the recipe→slot meal-eligibility check from scalar equality
    to set membership. It changes the `RecipeSnapshot` shape and the hard filter, but slot enumeration and
    determinism are untouched (see Phase 8).

### Decisions locked with the user
- **Scope**: a new **post-MVP "v2" epic**. The 26-Jun MVP boundary is unchanged; these items move out of OVERVIEW "Excluded/Future" into a planned v2 roadmap and are PRD'd now, built after MVP.
- **Recipe seeding**: **community shared + ranked** (publish + upvote + import-copy). Official starter pack = system-owned community recipes flagged `is_official`.
- **Food groups**: **hybrid** — static seed column on catalog ingredients + Claude-API fallback for user-created ingredients, cached on the row (first Anthropic SDK use in the repo, server-side only).
- **Inventory↔grocery**: **read-side, presentation-only annotation** — the grocery line always shows the full required quantity plus an on-hand annotation and a suggested "to buy" amount (e.g. "5 tomato — you have 2 · buy 3"). Never subtracts/hides the requirement, never mutates `grocery_items`.
- **Leftover spoilage is per-item**: `leftover_max_days` is only the *default*; each leftover row's `expiration_date` is independently editable, and the default is seeded from the ingredient's own `ingredients.max_storage_days` when present, else the workspace `leftover_max_days`.
- **Ingredient substitution is menu state, not a recipe edit and not inventory**: it lives in a per-slot override table consumed only by grocery recompute; it never edits the shared recipe, never feeds the engine, never changes `accepted_seed`.
- **Restrictions are split by mode**: **exclusive** = hard filter (existing behavior), **inclusive** = soft preference (new). Both are overridable per generation, including *relaxing* a profile restriction for a single generation. Inclusive prefs are implemented as a **deterministic** soft-bias in the engine, never a hard filter.
- **Multi-timeframe is a set, not a scalar**: a recipe's single `meal_type` becomes a *set* of eligible meal types, **stored as a `recipe_meal_types` junction table** (decided — mirrors `recipe_dietary_tags`, clean RLS via recipe ownership, indexable; the `meal_type[]` array column was the rejected alternative). The engine's meal check broadens from `==` to `includes`; slot enumeration (driven by member `meal_frequency`) is unchanged. Existing recipes backfill to a one-element set, so the no-preferences/no-change generation stays byte-identical.
- **Seeding** (refinement, not a feature): the current 30-item hardcoded `seed-ingredients` admin route stays the *pattern*, but the strategy moves to a **versioned, curated seed-pack manifest** (ingredients **and** a starter recipe set) that is internet-*sourced offline and reviewed*, then committed — **not** fetched live per new user. This dovetails with the community **official starter pack** (Phase A). The **hosted-deployment** path that consumes this pack is **v3** — see [v3.md](./v3.md).
- **AI is a separate v3 POC, gated after Phase 2**: the only AI use *inside v2* is the narrow, server-side food-group classifier (Phase 0). Everything smarter — auto-grouping, i18n (es/fr), voice/NL capture, metadata prefill — plus hosted deployment is tracked in **[v3.md](./v3.md)**, **not committed to v2 scope**.

## Part 2 — Implementation (build sequence)

Three independent tracks: **Track A (community recipes)**, **Track B (execution machine + menu/grocery view parity + ingredient substitution)**, and **Track C (preferences & generation-time overrides + multi-timeframe — the only engine work)** can run in parallel.

### Shared data-layer ritual (every phase)
New table → `cd packages/supabase && npx supabase migration new …` (deps order: `fn_`→`enum_`→`tbl_`+idx+trg→`rls_` enable→`rls_` policies). New column → use the **`supabase-add-column`** skill. After each migration: regen `database.types.ts`, update hand-written types in `packages/supabase/src/types/db.ts`, add `module/<table>.ts` + `<table>.react.ts`, export via barrel. Route handlers call CRUD fns directly. Delegate schema to **supabase-migration-author**, route handlers to **route-handler-engineer**, UI to **ui-component-builder**, tests to **vitest-integration-author**, engine work to **constraint-engine-engineer** + **determinism-snapshot-curator**.

### Phase 0 — Food groups (prereq for Phase 2)
- `ingredients` + `food_group text NULL`, `food_group_source` enum `{seed,ai,unset}`.
- Seed `food_group` for catalog ingredients (extend `apps/web/app/api/admin/seed-ingredients/route.ts` + `seed.sql`), `source='seed'`.
- `enum_metadata` label set `food_group` (via `sys_save_label`).
- `apps/web/lib/api/food-group-classify.ts` — Claude-API classify for user-created ingredients lacking a group, cache on row via `supabaseAdminClient`. Use the **claude-api** skill (prompt caching). Never in the engine.

### Phase 1 — Inventory (feature 1)
- `enum inventory_source {manual,purchase,leftover}`.
- `inventory_items` (`id, workspace_id, ingredient_id, source, quantity numeric CHECK>=0, unit, expiration_date date NULL, source_menu_id NULL, source_slot_id NULL, label text NULL, is_consumed bool, ts`). Indexes: `(workspace_id, ingredient_id) WHERE NOT is_consumed`, `(workspace_id, expiration_date) WHERE NOT is_consumed`. RLS read = active member, write = creator/admin (`fn_user_workspace_role`).
- `workspaces.leftover_max_days int NOT NULL DEFAULT 3` (`supabase-add-column`).
- Module `inventory-items.ts`/`.react.ts`. Partial-spoilage = plain `quantity` decrement (no event log in v2).
- Endpoints: `GET/POST /api/workspaces/[id]/inventory`, `PATCH/DELETE /api/workspaces/[id]/inventory/[itemId]`.
- UI feature folder `apps/web/app/(app)/inventory/` (page + `_components/`).

### Phase 2 — Shopping confirmation + food-groups (feature 2) — depends on 0,1
- `enum shopping_status {in_progress,complete,incomplete}`, `acquired_status {pending,acquired,partial,skipped}`.
- `shopping_sessions` (`id, menu_id, workspace_id, status, completeness numeric NULL, created_by, ts`; partial unique `(menu_id) WHERE status='in_progress'`).
- `shopping_item_status` (`id, session_id, grocery_item_id, acquired_quantity, status, UNIQUE(session_id,grocery_item_id)`).
- `apps/web/lib/api/shopping-finalize.ts` — quantity-weighted `computeShoppingCompleteness`; finalize spills purchased-but-unused into `inventory_items(source='purchase')`.
- Grouping = read-side `GROUP BY ingredients.food_group` (Phase 0); **does not** touch `recomputeGroceryListsForMenu`.
- Endpoints: `POST .../menus/[menuId]/shopping-sessions`, `GET .../shopping-sessions/active?group=food_group`, `PATCH .../shopping-sessions/[sid]/items/[groceryItemId]`, `POST .../shopping-sessions/[sid]/finalize`.

### Phase 3 — Incomplete alerts (feature 3) — DERIVED, no table — depends on 2 (reads 4 if present)
- `apps/web/lib/api/menu-alerts.ts` `deriveShoppingAlertsForMenu({admin,menuId})`: missing items (from `shopping_item_status`) → reverse `ingredient→menu_slot` map (via `recipe_ingredients`, **after applying ingredient overrides from Phase 6**) → alert per not-yet-cooked slot whose shortfall isn't covered by inventory. Degrades to "all slots planned" when cook-status absent.
- Endpoint `GET /api/workspaces/[id]/menus/[menuId]/alerts`. UI = badges on the menu/slot view (`apps/web/app/(app)/menu/_components/menu-view.tsx`). Rationale for no table: state is fully derivable and high write-amplification otherwise; repo has no server notification table today.

### Phase 4 — Cook-status (feature 4) — depends on nothing (sequenced after 3)
- `enum slot_cook_status {planned,cooked,skipped}`.
- **`slot_completions`** (`id, menu_slot_id, workspace_id, status, cooked_at NULL, notes NULL, UNIQUE(menu_slot_id)`) — a **separate table, not a column on `menu_slots`**, so cook-status is structurally invisible to `accepted_seed` and to slot-replace paths. Absent row = `planned`. RLS via `menu_slots→menus.workspace_id` EXISTS.
- Endpoint `PATCH .../menus/[menuId]/slots/[slotId]/completion {status,notes?}`.

### Phase 5 — Leftovers (feature 5, incl. per-leftover spoilage / item 7) — depends on 1,4
- Marking a slot `cooked` may emit `inventory_items(source='leftover', source_slot_id, source_menu_id, label, expiration_date = …)`.
- **Per-leftover expiry (item 7)** — `expiration_date` is computed **per leftover row**, not from one global knob:
  default = `cooked_at::date + COALESCE(ingredients.max_storage_days, workspaces.leftover_max_days)`
  (reuse the existing `ingredients.max_storage_days` column), and is then **independently editable** on the
  inventory row via the existing `PATCH /inventory/[itemId]`. No new schema beyond the per-row `expiration_date`
  already added in Phase 1.
- Consumption: manual `is_consumed=true` (or partial decrement) **or** lazy `expireLeftovers` on inventory read (no cron needed in v2); expiry is evaluated per row against its own `expiration_date`.

### Phase 6 — Menu-level ingredient substitution (item 9) — Track B, depends on accepted menus only
- **`menu_slot_ingredient_overrides`** (`id, menu_slot_id, workspace_id, original_ingredient_id, substitute_ingredient_id, quantity numeric NULL, unit NULL, note text NULL, created_by, ts`; `UNIQUE(menu_slot_id, original_ingredient_id)`). Keyed by `menu_slot_id` so it is **structurally unreachable from `accepted_seed`** (which hashes only recipe-tuples) and from the engine. RLS via `menu_slots→menus.workspace_id` EXISTS, write = creator/admin.
- `recomputeGroceryListsForMenu` (`apps/web/lib/api/menu-grocery.ts`) gains an override-application step: after loading `recipe_ingredients` for the menu's slots, left-join the override map and replace `(ingredient_id, quantity?, unit?)` per slot before aggregation. **Stays inventory-agnostic and engine-agnostic** — overrides are menu state. This is the one change to recompute's inputs; cover it heavily in the determinism regression (seed unchanged, grocery changed).
- Validation: a substitute is rejected (422) if it introduces an allergen or violates an exclusive restriction for that slot's eater(s) — reuse the engine's allergen/exclusion check (`packages/constraint-engine/src/filter.ts` `recipeViolatesAllergies` / `recipeHasExcludedIngredient` logic) at the route layer, mirroring the existing slot-replace validation in `apps/web/app/api/workspaces/[id]/menus/[menuId]/slots/[slotId]/route.ts`.
- Suggested substitutes come from the recipe's existing `recipe_ingredients.substitutions` JSONB catalog (currently unused) plus free choice from the ingredient catalog.
- Module `menu-slot-ingredient-overrides.ts`/`.react.ts`. Endpoint: `PUT|DELETE .../menus/[menuId]/slots/[slotId]/ingredient-overrides`. After a successful override write on an *accepted* menu, re-run `recomputeGroceryListsForMenu`.
- UI: substitution control on the slot detail / menu view (`apps/web/app/(app)/menu/_components/`), reusing the ingredient picker.

### Phase 7 — Menu/grocery view parity (items 8 + 10) — Track B, pure read-side, no schema
- **All-members aggregated grocery view (item 8)** — add a pure transform in `apps/web/lib/grocery-filter.ts`, e.g. `aggregateHouseholdGrocery({ lists })`, that unions the shared list + every per-member list into one consolidated list keyed by `(ingredient_id, unit)`, summing quantities (and folding `scheduled_purchase_day` to the earliest). Composes cleanly with `applyShopForFilter` and `annotateWithInventory`. Surface as a view mode in `apps/web/app/(app)/grocery/_components/shop-for-picker.tsx` (e.g. "Everyone" / "By member" / subset). No recompute or schema change.
- **Per-member menu switcher (item 10)** — mirror `shop-for-picker.tsx` on the menu screen: a member/household toggle on `apps/web/app/(app)/menu/_components/menu-view.tsx` that filters rendered slots by `target_member_id` (household = all). URL-param driven like `?shop_for=`. Pure read-side filter; no schema change. New small component `menu-member-picker.tsx` under the menu `_components/`.

### Phase A (parallel) — Community recipes (feature 6)
- Provenance on `recipes`: `source_publication_id NULL`, `imported_from_recipe_id NULL`, `imported_at NULL` (`supabase-add-column`; display-only, never engine input).
- `recipe_publications` (`id, recipe_id, workspace_id, published_by, is_official, is_unpublished, upvote_count, import_count, title_snapshot, cuisine_snapshot, ts`; partial unique `(recipe_id) WHERE NOT is_unpublished`; index `(is_unpublished, upvote_count DESC)`, `(is_official) WHERE is_official`).
- `recipe_upvotes` (`PK(publication_id,user_id)` → structural double-vote prevention; trigger maintains `upvote_count`).
- RLS: publications read `USING(NOT is_unpublished)` for authenticated + owner sees own; write = owning-workspace creator/admin. Upvotes write `WITH CHECK(user_id=auth.uid())`. **Additive** `recipes_read_published` policy (+ mirror on recipe sub-tables) `USING (is_deleted=false AND EXISTS(published, live))` — do **not** widen the existing `recipes_read`. (Template: `migrations/20260523000602_rls_create_recipe_policies.sql`.)
- Import = deep copy via `apps/web/lib/api/recipe-import.ts` reusing `createRecipe` (copies recipe + ingredients + instructions + dietary_tags; labels funnel through `sys_save_label`; global `ingredient_id` FKs need no copy); bump `import_count`.

### Track C — Inclusive preferences + generation-time constraint overrides (item 11) — THE ONLY ENGINE WORK
Sequence the schema/UI independently; the engine change is the gated piece (determinism snapshots).
- **Storage** — add inclusive preferences distinct from existing hard `member_dietary_restrictions` / `member_allergies`:
  `member_dietary_preferences` (`id, member_id, kind enum {dietary_tag,ingredient}, value text|ingredient_id, ts`; one row per liked tag/ingredient). (Alternatively a `preference_mode` enum on a unified table — chosen table form keeps the existing hard tables untouched.) RLS mirrors the existing member-constraint sub-tables.
  Endpoints mirror the existing `POST .../members/[memberId]/dietary-restrictions` family: `POST|DELETE .../members/[memberId]/dietary-preferences`.
- **Engine input** (`packages/constraint-engine/src/types.ts`) — extend `MemberSnapshot` with `dietaryPreferences: { tags: string[]; ingredients: string[] }` (inclusive). Extend `GenerateMenuOptions` with:
  `additionalDietaryPreferences?` (inclusive add at generation), and `relaxedDietaryRestrictions?` / `relaxedAllergies?` (subtractive — temporarily lift a profile *exclusive* restriction for this one generation).
- **Engine logic** — the soft-bias is the first real soft-constraint use of the deferred scoring hook in `packages/constraint-engine/src/assign.ts`. Among the **hard-valid** candidates for a slot, partition into "preferred" (matches an inclusive tag/ingredient) vs "rest"; the seeded RNG picks from "preferred" when non-empty, else "rest". This keeps generation **fully deterministic** (same inputs + seed = same output) and never excludes a valid recipe. `filter.ts` is unchanged for inclusive prefs (they are never hard filters); `filter.ts` honors `relaxed*` by removing those values from the hard sets *before* filtering.
- **Overlay** (`apps/web/lib/api/menu-overlay.ts` `computeEffectiveOverlay`, `apps/web/lib/api/menu-input-builder.ts`) — extend the additive dedup to also carry inclusive prefs, and add the subtractive `relaxed*` handling (remove from the member's effective hard set for this generation only). All of it lands in `inputs_hash` — a relaxed/preference-tuned generation is legitimately a *different* generation.
- **UI** — surface inclusive prefs on the member profile editor, and a per-generation override panel on the menu-generation form (add inclusive prefs, add/relax exclusive restrictions for this run). Reuse existing constraint pickers.
- **Determinism** — regenerate engine golden snapshots with **determinism-snapshot-curator**; the snapshot delta is *intentional* (new optional input fields + soft-bias selection). Prove: with no preferences/overrides, output is byte-identical to today; with preferences, output is stable across runs for a fixed seed.

### Phase 8 — Multi-timeframe recipes (item 12) — Track C / engine, gated on snapshots
