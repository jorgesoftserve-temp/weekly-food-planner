# v3 — AI Assist (POC) & Hosted Deployment

> **Status: research / proof-of-concept. NOT committed scope.** Everything here is **gated until after
> [v2](./v2.md) Phase 2 ships** and the v2 food-group classifier has proven the "Claude-on-server" pattern
> in production. This doc is the extraction of the v2 plan's v3 appendix (decided with the user) into its
> own roadmap, plus the hosted-deployment work needed to make any of it real.

## Context

Through [v1](./v1.md) (MVP) and [v1.5](./v1.5.md) (enhancements + agentic tooling) the product is a fully
deterministic, constraint-based planner — but it assumes a fairly motivated, technical user: someone who
types in recipes, ingredients, dietary tags, and meal types by hand, in English, on a local stack. v3 asks
a different question: **how much of that input can the system infer, so a lazy or non-technical user gets a
good plan with minimal typing — and can it run for real users on a hosted server?**

Two hard facts shape this plan:

1. **The repo has zero AI/i18n/web-fetch infrastructure today.** No `@anthropic-ai/sdk`, no Claude calls, no
   server-side WebSearch/WebFetch, no `next-intl` or locale files, no agent runtime. v2 introduces exactly
   **one** narrow server-side Claude use — the food-group classifier ([`food-group-classify.ts`], Phase 0).
   That is the beachhead; every v3 AI item extends from it.
2. **The repo is local-only.** No hosted-Supabase env, no production `.env`, no deploy step in CI
   (`.github/workflows/ci.yml` runs typecheck + test only). Nothing AI-for-real-users matters until there's a
   hosted target. So **hosted deployment is the true Phase 0 of v3.**

**Guiding constraint (inherited):** none of this may touch the constraint engine's determinism. AI suggests
*inputs* (recipes, tags, metadata, translated copy) that a human confirms; the engine still runs pure and
seeded on the confirmed data. AI is never in the generation path.

## Decisions / framing

- **Gate**: build nothing here until v2 Phase 2 is live and the food-group classifier is proven.
- **AI suggests, humans confirm**: every AI output is a *draft* the user accepts/edits — never auto-saved,
  never fed straight to the engine.
- **Sequencing**: hosted deployment first (enables everything), then auto-grouping (cheapest win off the v2
  beachhead), then the "smart capture" bundle (voice + metadata prefill), then i18n as a separate
  product-localization effort.
- **Cost discipline**: prefer one-time/cached classification over per-request calls; cache aggressively;
  prompt-cache the system prompt (as v2 already does).

## Phase D0 — Hosted deployment (prerequisite; ops, not AI)

Make the app real so AI features have users and a place to run.

- **Managed Supabase project** — provision; push the existing migration set with `supabase db push` (migrations
  are already the source of truth). Wire `SUPABASE_PROJECT_REF` + access token for the `supabase-remote` MCP
  server (already configured in [`.mcp.json`](../../.mcp.json), currently unused).
- **Env layout** — production `apps/web/.env.production` template; the service-role key stays server-only
  (keep the three-clients rule). Document secrets handling.
- **Seed-on-deploy** — run the admin seed route (or a migration-embedded `INSERT ... ON CONFLICT` of the v2
  seed-pack manifest) once against the hosted DB as a post-deploy step.
- **Deploy job in CI** — Vercel for [`apps/web`](../../apps/web) + a migrate/seed step. Absent today.
- Already designed deploy-aware in v2 (e.g. `ADMIN_API_KEY` required, stable system-workspace id).
- **Effort**: M (mostly config + one CI job). **Cost**: hosting + managed Supabase tier; no per-request AI cost.

## Phase D1 — Auto-grouping (lowest effort, highest confidence)

Group recipes/ingredients without user input — the direct extension of the v2 beachhead.

- Batch-classify **ingredients → food groups** (already partially in v2 for user-created ingredients) and
  **recipes → cuisine / candidate meal types / dietary tags** via a server-side Claude call, **cached on the
  row**. Reuses the v2 `food-group-classify.ts` shape and prompt-caching.
- Pairs naturally with v2 **item 12 (multi-timeframe)**: AI can *suggest* the set of meal timeframes a recipe
  fits; the user confirms.
- **Missing tooling**: nothing beyond the `@anthropic-ai/sdk` already added in v2.
- **Cost**: per-row, one-time, cached — negligible. **Effort**: S.

---

## Phase D2 — Smart capture (voice + metadata prefill)

The "lazy / non-technical user" bundle. Two features, one shared pattern: *Claude structured-output →
existing recipe form*, always confirmed by the user.

- **Voice / natural-language capture** — transcription is the **Web Speech API** (client-side, free, already
  on phones/laptops). AI's value-add is *parsing* free-form dictation ("chicken sandwich, two slices of
  bread, tomato, lettuce, takes ten minutes") into the structured recipe shape (name, ingredients[],
  instructions[]). A server-side Claude structured-output endpoint feeds the existing
  [`recipe-form.tsx`](../../apps/web/app/(app)/recipes/_components/recipe-form.tsx).
  - **Missing**: a speech-to-form parsing endpoint; graceful fallback when the browser lacks the API.
  - **Cost**: per-dictation parse. **Effort**: M.
- **AI metadata prefill** — when a user types a recipe name the DB doesn't have, suggest `cuisine`,
  `prep/cook time`, `calories_per_serving`, `dietary_tags`, even a draft ingredient/instruction list (the 10
  scalar fields + 3 arrays the recipe form already validates). Server-side Claude structured output,
  **suggested, never auto-saved**.
  - Synergizes with v2 **community recipes**: prefer importing a real published recipe over generating one.
  - **Missing**: a "suggest metadata" endpoint + UI affordance; optional WebSearch grounding for real-world
    values (no server WebSearch tooling exists today).
  - **Cost**: per-suggestion. **Effort**: M.

## Phase D3 — Localization (i18n: Spanish + French)

The heaviest item — and the least "AI": it's cross-cutting copy work with AI as an accelerator.

- Introduce `next-intl` (or equivalent) + a message catalog; extract every hard-coded UI string.
- AI translates **static UI strings** (review-then-commit, like the seed pack) and optionally **user content**
  on demand (cache aggressively).
- **Missing tooling**: i18n framework, locale routing, a translation pipeline/cache, an RTL-readiness review.
- **Cost**: static strings = one-time; on-demand user-content translation = per-request (cache). **Effort**: M–L
  (touches every component's copy). Scope as a standalone product-localization effort, not bundled with the
  AI capture work.

## Phase D4 — Further UX ideas (enumerate, don't commit yet)

- **Natural-language menu generation** — "a cheap vegetarian week, no nuts" → fills the v2 item-11/Track C
  generation overrides (inclusive prefs + relaxed/added exclusive restrictions). The engine still runs
  deterministically on the resolved inputs.
- **Chat assistant over the `menu` MCP tools** — the workspace/engine tools (`engine_generate_menu`,
  `workspace_preview_menu`, `workspace_member_constraints`, `workspace_recipe_usability`, `workspace_recent_menus`)
  already exist as an agent surface; a conversational layer could drive them. Needs an agent runtime (none today).
- **Smart leftover → recipe suggestions** — "you have rice + chicken → make…", reading the v2 inventory.
- **Shopping-list OCR / receipt import** — auto-fill the v2 shopping session from a photographed receipt
  (vision model). Net-new tooling.

## Complexity, missing tooling & recommendation

| Item | Effort | New tooling needed | Cost shape |
|---|---|---|---|
| D0 Hosted deploy | M | managed Supabase, Vercel, CI deploy job | hosting (no AI cost) |
| D1 Auto-grouping | S | none beyond v2's SDK | one-time, cached — negligible |
| D2 Voice + prefill | M | structured-output endpoints; Web Speech fallback | per-call |
| D3 i18n es/fr | M–L | `next-intl`, locale routing, translation cache | one-time + cached per-request |
| D4 NL gen / chat / OCR / leftovers | varies | **agent runtime, server WebSearch/WebFetch, vision** (none today) | per-call |

**Missing across the board**: an agent/runtime layer, server-side WebSearch/WebFetch grounding, and an i18n
framework — none exist today.

**Recommendation**: ship **D0** to make the app real, treat **D1** as a fast-follow on v2 (near-free), bundle
**D2** as the actual "smart capture" POC that demonstrates the lazy-user value, and scope **D3** separately.
Defer **D4** until D1–D2 prove the suggest-confirm UX. Build none of it until v2 Phase 2 ships.

## Verification (when/if built)

- **Determinism untouched**: confirm no AI output reaches the engine directly — AI fills *form drafts*; the
  engine still runs pure/seeded on user-confirmed data. The v2 determinism regression must still pass unchanged.
- **Suggest-confirm**: every AI-suggested field is editable and requires explicit user save; nothing auto-persists.
- **Cost guards**: classification cached on the row (no repeat calls); per-request features rate-limited and cached.
- **Hosted smoke**: migrations apply cleanly to the managed project; seed-on-deploy is idempotent; the
  end-to-end flow ([`scripts/verify-flow.mjs`](../../scripts/verify-flow.mjs)) passes against the hosted target.
- **i18n**: snapshot the message catalog; verify no untranslated keys in es/fr; locale routing round-trips.
