## Context

The product assumes a motivated, technical user: someone who types in recipes, ingredients, dietary tags,
and meal types by hand. v3 asks a different question: **a user already has a list — a guide from their
nutritionist, or their own weekly plan — so why re-type it?** v3 lets them hand that list to the app, which
**reviews what the workspace already has, figures out what's being asked for, and generates a menu plus the
backing recipes** to support it.

To make the input easy, we ship a **fillable template** (an `.md` and/or `.xlsx` the user copies their
dietary restrictions / desired meals into) generated by an improved **Claude skill**, and the importer
accepts **free-form pasted lists** too. Input may be **Spanish or English**, and — since the product should
read in the user's language end-to-end — v3 also makes the **app fully localized (i18n: ES + EN)**.

This is the first real AI surface in the product. It stands on the v2 beachhead: the narrow server-side
food-group classifier ([`food-group-classify.ts`], v2.0 Phase 0) proves the Claude-on-server + prompt-cache
pattern; v3 extends it from "classify one field" to "parse a whole list and assemble a plan."

## The one hard rule (inherited, non-negotiable)

**AI suggests inputs a human confirms; the constraint engine still runs pure + seeded on the confirmed
data.** No AI output reaches the engine directly, and **post-accept state never changes `accepted_seed`**.
Concretely:
- Every AI-parsed recipe and every AI-assembled menu is a **draft** the user reviews/edits and explicitly
  saves. Nothing auto-persists; nothing is generated against until confirmed.
- v3 may shape **generation inputs** (preferences, relaxed/added restrictions) — that is *allowed* to change
  `inputs_hash`, exactly as the v2.1 overlay already does — but the determinism contract (*same inputs +
  same seed = same output*) holds, and any engine-touching shift batches with an intentional golden-snapshot
  regen via **determinism-snapshot-curator**.

## Decisions / framing
- **AI suggests, humans confirm** (above) — the spine of the whole feature.
- **Cost discipline** — prompt-cache the system prompt (as v2.0 does); parse is per-import (cache where the
  same template is re-submitted); classification is cached on the row.
- **Reuse, don't reinvent** — recipe materialization is the **v2.1 bulk-create primitive**; menu assembly is
  the **v2.1 item-11 generation overlay**; recipe auto-tagging reuses the **v2.0 classifier**.
- **i18n is part of v3, not deferred** — the import understands ES/EN, so the UI should too.

## Phase v3.0 — Import template + Claude skill + parser

The input pipeline: give the user an easy way to express what they want, and turn it into structured data.

- **Import-template Claude skill (dev-time).** A new skill under [`.claude/skills/`](../../.claude/skills/)
  that emits the **user-facing fillable template** — an `.md` and/or `.xlsx` layout covering: members +
  their dietary restrictions (exclusive) and preferences (inclusive), desired meals / recipes, meal
  timeframes, and the week shape. This is the "improve our Claude skills so we can generate an
  easy-to-understand template" ask. The emitted artifact is what we hand to users; the skill keeps it in
  sync as the recipe/constraint shape evolves.
- **Parser endpoint (server-side, hosted).** A Claude **structured-output** endpoint that ingests a filled
  template **or** a free-form pasted list (**ES or EN**) and returns the structured shape the app already
  validates: recipe drafts (the 10 scalar fields + 3 arrays the recipe form validates) and a menu request
  (members, timeframes, preferences/restrictions). Reuses the v2.0 Claude-on-server + prompt-cache pattern.
  *(Absorbs the old D2 "parse free-form → recipe form" half + the old D1 auto-grouping: parsed recipes are
  auto-classified — cuisine / candidate meal timeframes / dietary tags / food groups — via the v2.0
  classifier, all as suggestions.)*
- Graceful failure: an unparseable line surfaces as an editable "couldn't read this — fix it" row, never a
  silent drop.

## Phase v3.1 — Workspace-aware generation (the "chat")

Turn the parsed request into a real menu + recipes, with the workspace's existing data in the loop.

- **"Review what the workspace already has" = dedup/match.** Before creating anything, match parsed items
  against existing recipes/ingredients in the workspace and **suggest reuse vs. create-new** (e.g. "you
  already have *Chicken sandwich* — use it?" / "this *tomato* maps to your catalog ingredient"). The user
  resolves each.
- **Materialize accepted new recipes** via the **v2.1 bulk-create primitive** (one transactional insert of
  N recipes + children) — v3 does not get its own insert path.
- **Assemble the menu** by feeding the resolved request as **generation overrides** (inclusive preferences +
  relaxed/added exclusive restrictions) into the **deterministic engine** through the **v2.1 item-11
  overlay**. The engine still runs pure/seeded on the resolved inputs; AI only shaped the inputs.
  *(Absorbs the old D4 natural-language menu generation + chat-over-`menu`-MCP-tools idea — the existing
  `engine_generate_menu` / `workspace_preview_menu` / `workspace_member_constraints` /
  `workspace_recipe_usability` / `workspace_recent_menus` tools are the surface a conversational layer
  drives.)*
- **Suggest-confirm UI** throughout: every generated recipe and the assembled menu are editable drafts
  requiring an explicit save/accept. Accept routes through the normal menu-accept path, so `accepted_seed`
  derives from confirmed slot tuples exactly as today.

## Phase v3.2 — Full app i18n (ES + EN)

The heaviest sub-phase — cross-cutting copy work, AI as accelerator.

- Introduce `next-intl` (or equivalent) + a message catalog; extract **every** hard-coded UI string.
- ES + EN locales; locale routing that round-trips; a translation review pipeline (AI translates static UI
  strings, review-then-commit — the seed-pack discipline applied to copy).
- Optional, deferred-within-phase: on-demand translation of **user content** (recipe names/notes), cached
  aggressively. *(Absorbs the old D3.)*
- Pairs with the importer understanding both languages, so the product is coherent end-to-end in either.

- **Skill** — `.claude/skills/<import-template>/SKILL.md` + the emitted `.md`/`.xlsx` template artifact.
- **Endpoints** — `POST .../workspaces/[id]/import/parse` (list/template → structured drafts) and the

- **Stack** — `@anthropic-ai/sdk` (already added in v2.0, server-only, prompt-cached); `next-intl` for i18n.
- **Reuse** — `food-group-classify.ts` pattern (v2.0), the bulk-create primitive + item-11 overlay (v2.1),
  the existing `recipe-form` validation, the `menu` MCP tools as the agent surface.
- **No new engine-feeding tables** — the import writes ordinary recipe rows via bulk-create; any user-locale
  preference is profile/post-accept state, never an engine input.

## Determinism
v3 changes generation **inputs** only, and only at generation time (via the v2.1 overlay) — never
post-accept state, never `accepted_seed`. Engine purity holds. If a parse/override path is found to alter
engine inputs in a new way, batch the change with an intentional golden-snapshot regen
(**determinism-snapshot-curator**); the import path itself (recipe creation) is engine-free.

## Verification (when/if built)
- **Determinism untouched** — confirm no AI output reaches the engine directly: AI fills *drafts*; the engine
  runs pure/seeded on user-confirmed data. The v2 determinism regression passes unchanged.
- **Suggest-confirm** — every AI-suggested recipe/menu field is editable and requires explicit save; nothing
  auto-persists.
- **Dedup** — a parsed item that matches an existing recipe/ingredient offers reuse; accepting reuse creates
  no duplicate row.
- **Bilingual parse** — the same nutritionist list in ES and in EN produces equivalent structured drafts.
- **Cost guards** — system prompt prompt-cached; classification cached on the row; parse cached per identical
  template submission.
- **i18n** — snapshot the message catalog; no untranslated keys in ES/EN; locale routing round-trips; the
  end-to-end flow ([`scripts/verify-flow.mjs`](../../scripts/verify-flow.mjs)) passes in both locales.
