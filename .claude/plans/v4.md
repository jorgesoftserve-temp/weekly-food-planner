## Context

Community recipes lets users **publish/rank/import recipes cross-workspace so new users don't re-enter
basics.** It was originally item 6 of the [v2 epic](./v2.md) and is now its own version because it is too
large for one release and is **orthogonal to v2's execution/pantry spine and v3's AI import**.

The defining decision: **there is no "collaborative" (shared-mutable) ownership in the system** — community
is **copy semantics** end to end (publish → others import a *duplicate* into their own workspace; each import
is an independent recipe the importer owns). This slots under the existing workspace-scoped, single-owner
RLS model as an **additive** published-read policy; the existing `recipes_read` is never widened.

The full flow the user described breaks into three slices:

- **[v4.0 — Share & duplicate](./v4.0.md)** *(the headline community goal)*: a user shares a workspace recipe
  to the community; others import a **deep-copy duplicate** into their workspace (provenance + lineage
  tracked). If the owner deletes a shared recipe, they choose whether to also remove it from the community —
  in which case existing imported copies simply **become ordinary non-shared recipes** in the other
  workspaces (the copies persist; only the community link drops). Includes the **official starter pack** +
  the versioned seed-pack manifest (same publish/import mechanism). *(Recipe version "see original / revert
  vs keep current copy" is a flagged sub-feature — candidate for its own slice, needs version snapshots.)*
- **[v4.1 — Reviews & voting](./v4.1.md)**: upvote/downvote, plus a lightweight **review + comment** system
  (not just a vote), and the ranking signals those produce.
- **[v4.2 — Community page](./v4.2.md)**: the discovery surface — browse + **filter by popularity /
  most-shared / review ranking** — and a **"My community"** tab (recipes I shared + recipes I imported).
  *(Tab UX to be designed when we get there.)*

## Dependencies

- **Reuses the v2.1 bulk recipe-create primitive.** The deep-copy import is "bulk-create + provenance," not
  a fresh insert path — the transactional N-recipe insert ships in [v2.1](./v2.1.md) (also used by
  [v3](./v3.md)). v4.0 layers lineage/attribution on top of it.
- **Builds on hosting (v2.3).** Cross-workspace community is most meaningful on the hosted, multi-user
  deployment that [v2.3](./v2.3.md) stands up.

## The one hard rule that shapes the whole design

**None of the community/provenance state may feed the constraint engine or change `accepted_seed`** — the
engine stays pure and deterministic. `lineage_id`, `original_creator_id`, publications, votes, and reviews
are **display/provenance/community state only, never engine input**. An imported recipe yields the same menu
as its source under identical constraints. **No snapshot regen anywhere in v4.**

## Release sequencing

The cutline is dependency order, not the engine boundary (the whole track is engine-free):

| Slice | Feature | Release | Depends on | Rationale |
|---|---|---|---|---|
| 6a | Share + import-duplicate (+ official pack + seed-pack) | **[v4.0](./v4.0.md)** | v2.1 bulk-create, v2.3 hosting | publish + deep-copy import + provenance + owner-delete-unlink; cross-workspace **published-read** RLS (additive, never widens `recipes_read`) |
| 6b | Reviews + voting | **[v4.1](./v4.1.md)** | v4.0 publications + lineage | `recipe_upvotes` (one vote/user) + reviews/comments + ranking signals |
| 6c | Discovery page + "My community" | **[v4.2](./v4.2.md)** | v4.0 + v4.1 | browse + filter (popularity / most-shared / review rank); personalized tab |

**Test coherence:** the community track owns the **cross-workspace published-read RLS** — new territory the
rest of the product never exercises (everything else is single-workspace-scoped). Each slice's
integration suite (publisher-in-A / importer-in-B role matrix) is authored with **vitest-integration-author**.

**Seed-pack seam:** the **versioned manifest** (`packages/supabase/seed-pack/{ingredients,recipes}.json`)
lands with v4.0's official starter pack (same publish/import mechanism). v2.0's Phase 0 keeps the existing
inline 30-item seed array as the interim pattern until then — no orphaned dependency.
