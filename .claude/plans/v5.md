# v5 — Native mobile app (Expo, phased)

> Roadmap: **[v2](./v2.md)** → **[v3](./v3.md)** → **[v4](./v4.md)** → **v5** (this — native mobile) →
> **[v6](./v6.md)** (deeper/ambient AI).
>
> **Status: planned, deferred.** Not on the critical path: the hosted web app ([v2.3](./v2.3.md)) already
> serves mobile/tablet viewports responsively, so **beta users get a mobile-capable experience without a
> native app**. v5 is the genuine native client, built only after the web product (v2–v4) is proven.

## What v5 is

A **full native Expo (React Native) client** under `apps/mobile/`, delivered as a **phased track**
(`v5.0 … v5.n`). The driving constraints, locked with the user:

1. **Store-rule-compliant per phase** — every phase produces a build that could pass App Store / Play Store
   review (in particular Apple guideline **4.2 minimum functionality** — *not* a thin WebView wrapper).
2. **Never block web development** — mobile is its own app workspace + shared packages; web releases
   continue in parallel, and mobile phases interleave with other work.
3. **One codebase, not two** — maximize shared code/components/utilities. The UI layer is the only thing
   that is genuinely platform-specific; everything below it (data, validation, types, business logic, design
   tokens) is shared.

This is the single largest item in the roadmap — sized like a **second front-end**, not a sub-release. The
exact phase count is refined when the track starts.

## Phase v5.0 — Shared-code foundation + store-compliant native shell

The prerequisite that makes "one codebase" real *before* any screen is ported. Two halves:

**Shared-code extraction.** Pull the platform-agnostic layers into shared packages both `apps/web` and
`apps/mobile` consume:
- `packages/supabase` — **already shared** (data layer + module hooks). Adapt the client construction for RN
  (the three-clients discipline holds; browser cookie storage → RN secure storage for the auth client).
- `packages/constraint-engine` — **already pure + shared**, no change.
- **New / expanded shared packages** for **TypeScript types, Zod validation schemas, and pure utilities** —
  move UI-free web helpers (e.g. the grocery-filter logic in `apps/web/lib/grocery-filter.ts`) into a
  platform-agnostic package so both clients call the same code. Audit `apps/web/lib` for anything that is
  pure logic masquerading as web code.

**Native shell.**
- Stand up `apps/mobile/` on **Expo + Expo Router**; wire the Supabase RN client + native auth (secure
  token storage, deep-link auth callback).
- **NativeWind** so the **design-system-architect's Tailwind tokens reuse cross-platform** — the web
  `globals.css` token values become a shared token source consumed by both web and RN via the same
  `className` API. Leaf primitives still differ (shadcn web vs RN components), but the *design language*
  (colors, spacing, radii, accent) is one source of truth, owned by **design-system-architect**.
- Ship a **genuinely native** first build: native auth + nav shell + 1–2 real native screens (e.g. the menu
  view + grocery view), enough to clear store minimum-functionality review.

## Phase v5.1+ — Incremental screen port

One phase per screen cluster, each reusing the shared hooks + validation, each independently releasable:
- menu view → grocery list → recipes CRUD → members / settings → (community, if shipped).
- Each phase swaps a cluster from "not yet in the native app" to "native screen on shared data." The web app
  is never paused; phases slot between other v-line work.
- Per-phase: **ui-component-builder** (RN screens), **design-system-architect** (token parity), pre-ship
  **accessibility-auditor** (native a11y) + **ux-reviewer** passes.

## Out of scope
- Re-architecting the web app — web stays Next.js; v5 only *extracts shared logic* it already contains.
- A separate mobile backend — mobile hits the same hosted Supabase (v2.3) through the same RLS.

## Determinism
Engine-free, schema-free. The native client consumes the **same** `packages/constraint-engine` and
`packages/supabase`; it changes *where* the UI renders, not *what* the engine computes. No snapshot regen.

## Verification (per phase)
- **Shared code is actually shared** — `apps/web` and `apps/mobile` import the same engine/validation/util
  packages; no logic is duplicated between them (lint/dep-graph check).
- **Store compliance** — each phase's build passes a store-readiness checklist (native auth, native nav,
  real native screens, no bare WebView shell).
- **Token parity** — a `design-parity` pass confirms the native screens match the web design tokens
  (colors/spacing/radii/accent) in light + dark.
- **Web unaffected** — `pnpm typecheck && pnpm test` for `apps/web` stays green throughout; the web release
  cadence is not gated on mobile.
- **End-to-end** — auth → generate → view menu/grocery works natively against the hosted backend.
