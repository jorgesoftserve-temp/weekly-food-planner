# supabase-add-column

Adding a column in this repo is a multi-file change that's deceptively easy to get wrong: the migration goes in one place, the partial-unique index covers `is_deleted = false`, the types regen has to run and the generated file consumed by every module reading the table, the `<table>.ts` `*_SELECT` constant has to include the column, the `<Table>Record` type has to gain the field, any `Create*Payload` / `Update*Patch` types must follow suit, the Zod schemas on the route handler need to accept the new field, and the integration test should cover the new behaviour. This skill walks the change in order and emits every artifact.

## When to invoke

- Adding a single scalar column (`text`, `int`, `numeric`, `boolean`, `timestamptz`, `jsonb`, an existing enum, a `uuid` FK, etc.).
- Adding a small **set of related columns** in the same migration (e.g. `cost_per_unit` + `cost_currency`).
- Adding a column that requires a backfill from existing data.

## When NOT to invoke

- Creating a brand-new table → use [`supabase-migration-author`](../../agents/supabase-migration-author.md) agent.
- Creating an RPC or stored function → `supabase-migration-author` agent.
- Adding a new junction table → `supabase-migration-author` agent.
- Renaming or dropping a column (different risk profile; needs explicit human review).
- Changing a column's type (data-loss risk; needs explicit planning).
- Adding RLS policies that aren't directly tied to the new column → `supabase-migration-author` agent.

If the change touches more than columns, this skill emits the column part and hands the rest off in the report.

## Input

The user supplies (or the skill asks for, one round of clarification only):

```yaml
table: snake_case_plural          # e.g. ingredients, workspace_members
columns:
  - name: snake_case               # e.g. cost_per_unit
    type: text | int | numeric | boolean | timestamptz | jsonb | uuid | <enum_name>
    nullable: true | false
    default: <literal or null>     # SQL literal, e.g. 'usd', 0, true, "'{}'::jsonb"
    references?:                   # for FK columns
      table: snake_case_plural
      column: id
      onDelete: CASCADE | SET NULL | RESTRICT
    comment: "one-line column comment for COMMENT ON COLUMN"
backfill?:                         # if existing rows need values
  description: "what we're setting and why"
  sql: |
    UPDATE public.<table>
    SET <column> = ...
    WHERE ...
indexImpact?:                      # list of existing partial indexes the new column should be added to
  - name: uq_<table>_<predicate>
    addToPredicate: true
extensibleLabel?:                  # if the column stores a value from enum_metadata (cuisine_type, dietary_restriction, dietary_tag, food_allergy)
  enumType: cuisine_type | dietary_restriction | dietary_tag | food_allergy
```

If the user just describes the change in prose, ask **once** for: (a) the target table, (b) the column name + type + nullable/default, (c) whether existing rows need a backfill, (d) is it an extensible-label column. Then proceed.

## Steps

1. **Confirm the table exists** — `Glob` for `packages/supabase/supabase/migrations/*tbl_create_<table>*.sql`. If not found, abort with a clear message ("Table `<x>` doesn't exist; use `supabase-migration-author` agent to create the table first").
2. **Pick the migration filename.** Pattern: `[timestamp]_tbl_<table>_add_<column>_<modifier>.sql`. Modifier examples: bare for a simple add, `_with_backfill` if backfill SQL is included, `_with_index` if the new column gets its own index. Use the current UTC date and a sub-second slot that doesn't collide with existing migrations.
3. **Detect partial-index impact.** Read the table's create migration. Note partial unique indexes filtering on `is_deleted = false`. If the new column is `NOT NULL` and changes uniqueness semantics, the indexes need to be rebuilt — flag it.
4. **Emit the migration SQL.** Sections in this order:
   - Header comment block: what's being added, why, link to PRD section, backfill behaviour (if any).
   - `ALTER TABLE public.<table> ADD COLUMN <name> <type> [NOT NULL] [DEFAULT <literal>];`
   - One `ALTER TABLE` per column when adding multiple — preserves readability.
   - `COMMENT ON COLUMN public.<table>.<name> IS '<comment>';` for every column.
   - Backfill block: `UPDATE ... SET ... WHERE ...;` with a comment explaining the rule. Always idempotent (`WHERE <new_col> IS NULL` for nullable adds, or scoped by some other condition).
   - Index changes (`DROP INDEX IF EXISTS` + recreate, or `CREATE INDEX` for new ones) — only if the column needs an index, the table has a soft-delete partial index it affects, or the column is part of a join hot path.
5. **Emit the types regen command.** Always `pnpm --filter @weekly-food-planner/supabase db:gen:types`. Note that the dev DB must be running (`pnpm --filter @weekly-food-planner/supabase db:start`).
6. **List the module TypeScript files to touch** (read `packages/supabase/src/module/<table>.ts`):
   - `<Table>Record` type — add `<column>: <ts-type>` (or `<ts-type> | null` if nullable, no default).
   - `<TABLE>_SELECT` constant — append `, <column>` so the list query returns it.
   - `Create<Table>Payload` — add `<column>?: <ts-type>` if optional on create, required if `NOT NULL` without default.
   - `Update<Table>Patch` — add `<column>?: <ts-type>` (patches are always partial).
   - `create<Table>` / `update<Table>` functions — add the field to the insert/update payload mapping.
   - If extensible-label: route the value through `sys_save_label(<enum_type>, value)` RPC before write. See [`packages/supabase/src/module/labels.ts`](../../../packages/supabase/src/module/labels.ts) for the existing pattern.
7. **List the route handler files to touch** — `Glob` for `apps/web/app/api/.../route.ts` that import from `<table>.js`. For each:
   - Zod schema (`schema.ts` next to the route or inline) gains a parser for the new field.
   - Response shapes return the new field.
   - If the column is role-gated (e.g. only admins can set it), add the server-side check explicitly — RLS is the safety net but the 403 should be clear.
8. **List integration tests to update** — `Glob` for `*.integration.test.ts` referencing the table. Recommend a new test covering: happy path (column persists), backfill (existing rows have the right value), nullable handling (default applies), and RLS if the column changes visibility.
9. **PRD update.** Locate the §6.x sketch for the table in [DATABASE_PRD.md](../../../docs/PRD/DATABASE_PRD.md). Emit the suggested patch (the new row in the column table + notes if the rules above interact with §6.16 soft-delete or §6.11.1 `generation_options` shape).
10. **Report** in the structure below.

- ...

## Non-negotiables

- **Always use the `db:migration:new` script.** Never hand-write the migration filename or its timestamp.
- **Idempotent backfill.** Backfill SQL must be safely re-runnable (`WHERE <col> IS NULL`, or scoped by some other stable condition). Re-running a migration mid-development should not double-apply.
- **`COMMENT ON COLUMN` is mandatory.** Every new column gets a one-line comment. The PRD references it; absence makes future agents guess intent.
- **Partial-index awareness.** If the table has `is_deleted = false` partial unique indexes (most do), confirm the new column doesn't change their semantics. If it does (e.g. introduces a new uniqueness dimension), flag explicitly.
- **`NOT NULL` without `DEFAULT` is a hard fail on existing tables.** Either provide a `DEFAULT`, make the column nullable, or backfill in the same migration AND then `ALTER COLUMN ... SET NOT NULL`. The skill should emit all three steps in that order.
- **Types regen happens before TS edits.** The module TS edits assume the new `database.types.ts` is on disk. The report orders commands accordingly.
- **No drive-by changes.** This skill only emits column adds. If reading the table reveals an unrelated issue (missing index, wrong nullable), surface it in the report but do not include it in the migration.

## What to flag in the report

- **Backfill that requires data the migration doesn't have.** E.g. `cost_per_unit` for `ingredients` needs price data; the migration can default to `NULL` but the feature is meaningless until the catalog is seeded. Say so up front; recommend a follow-up service-role seed.
- **Extensible-label columns** that need `sys_save_label` plumbing on the write path. The migration is just `text` — the runtime guarantee comes from the RPC. Cross-reference [DATABASE_PRD §10](../../../docs/PRD/DATABASE_PRD.md) and the `labels.ts` module pattern.
- **Columns referenced by RLS policies.** If the new column will be read by an existing policy (e.g. `role`-gated visibility), the policy update must ride the same migration or follow immediately — flag if you can't tell.
- **Partial-index reshape** if the new column changes uniqueness. The reference migration `20260526000100_tbl_menus_add_draft_accept_columns.sql` is the canonical example of dropping + recreating partial unique indexes in the same file.
- **Multi-module consumers.** If `Glob` finds more than ~3 files importing the table's `Record` type, list them all — the skill emits the patch for the module file, but app-side TypeScript may need follow-up.
- **PRD §6.11.1 / `generation_options` shape.** If the new column is on `menus` and is part of the engine's audit snapshot, the `generation_options` JSON example in DATABASE_PRD §6.11.1 also needs updating. The skill should detect this and include the doc patch.
